# v1.18 — Native-enforcement fallback: no rule left unenforced when a tool is absent (2026-06-30)

Operator directive: *"If a tool cannot be found for a rule, it should run on the native enforcement.
No rule can be unenforced."* This refines (does not contradict) the §28.28 missing-tool policy.

## The gap

`rubric/composite-dispatch.sh` routed a file to its 3rd-party tools and, when every routed tool was
absent/unadapted, reported `status=incomplete / not_enforced=N` — leaving the file's rules unenforced.
Meanwhile the dependency-free native detectors (`rubric/enforce-file.sh`) already produce a real verdict
on the same file (e.g. catching a `0.0.0.0/0` ingress as a P0 fail with checkov absent). The routed path
simply did not fall back to them.

## The change (Phase 1)

`composite-dispatch.sh` now falls back to native enforcement (`enforce-file.sh`) whenever the routed
tool path cannot produce a verdict:

- **No applicable tool** (the file's kind maps to no routed tool) → native enforcement instead of a
  vacuous green.
- **Every routed tool absent/unadapted** (no tool produced green or red) → native enforcement instead
  of `incomplete`.

Verdict folding: native blocking → `red`; native clean/advisory → `green`; native cannot verify
(exit 3) → `incomplete` (the honest floor — never a vacuous green). New markers:
`dispatch native-fallback file=<f> verdict=<green|red|incomplete> rules_checked=<n>` and `fallback=native`
on the summary line.

**§28.28 hard-require is preserved:** an explicitly `--required` tool that is absent still hard-fails to
`red` *before* any fallback (CTP must not claim a gate it cannot run). The native fallback applies only
to the optional/absent path that previously degraded to `not_enforced`.

**Tool-present path is byte-identical:** when a routed tool is present and produces a verdict, the
fallback never runs.

The invariant now holds across all enforcement surfaces: write-time (`enforce-standards-on-save.sh` →
`enforce-file.sh`, native always runs), audit-time (`composite-audit.sh` already invokes `enforce-file.sh`
per file), and standalone routed dispatch (now falls back).

## Phase 2 (separate section) — universal native enforcer

Phase 1 routes to the native enforcer; Phase 2 makes the native enforcer *capable of enforcing any*
software-engineering / software-architecture rule (including scraped rules whose only `enforced_by` is
an absent 3rd-party tool and that carry no bespoke detector) by falling through to the universal
semantic-projection detector (`prose-judge.sh`, CTP-D-3). Documented in its own amendment.

## §20 note

Enforcement-robustness refinement over the existing composite engine; preserves the §21
definition-of-done (the gate is unchanged; a previously-unenforced path now enforces). Additive — no
existing §1–§28.55 content altered.
