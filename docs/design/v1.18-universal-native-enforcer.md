# v1.18 — Universal native enforcer: any rule enforceable natively when no tool is found (2026-06-30)

Operator directive (Phase 2, follows §28.56): *"The native enforcer must be capable of enforcing any
rule to software engineering and software architecture when a 3rd-party tool cannot be found to enforce
the rule."*

## The gap

§28.56 makes the routed dispatch path fall back to the native enforcer (`enforce-file.sh`). But the
native enforcer only enforced rules that carry a **bespoke deterministic detector** (line: `next unless
r["id"] && r["detector"]`), plus `applies_to_prose` rules via prose-judge **only on `.md`**. A rule with
no native detector — e.g. a scraped rule whose only `enforced_by` binding is a 3rd-party tool — was
silently skipped when that tool was absent, and a rule whose detector file could not be found was
skipped too (`next unless File.exist?(detp)`). So "no rule unenforced" was not actually guaranteed for
detector-less / tool-only rules.

The architecture already ships the universal enforcer: **`prose-judge.sh` (CTP-D-3)** — a
semantic-projection detector that takes *any rule body + any content* → violates / compatible / abstain,
three tiers (deterministic keyword → LLM under `LLM_JUDGE=1` → not_enforced floor). It just wasn't wired
as the universal fallback, and it could only judge rules already in the catalog.

## The change

**`prose-judge.sh`** gains an inline rule interface:
- `--body <text>` supplies the rule body directly (instead of resolving the id from the catalog), so it
  can judge ANY rule — including a scraped rule not (yet) in the catalog.
- `--forbid <csv>` supplies operator literal forbidden tokens; these are matched as substrings even when
  they contain regex metacharacters (e.g. `eval(`), unlike catalog `patterns`.

**`enforce-file.sh`** now routes every applicable rule that lacks a runnable deterministic detector
through the universal enforcer:
- A catalog/extra rule with **no `detector`** (e.g. `enforced_by` a 3rd-party tool only) → `universal`
  set → enforced via `prose-judge --body <description> [--forbid <token>]` on the file.
- A detector-bearing rule whose **detector file cannot be found** → same universal fallback instead of a
  silent skip.
- The universal enforcer runs on **any file type** (not just `.md`).
- A rule with no scope glob applies universally to the file (never silently skipped).
- A rule the enforcer cannot deterministically judge (no token, no model) surfaces as `not_enforced`
  (the honest floor — never a silent green).
- `--extra-rules <yaml>` (operator/test affordance) merges ad-hoc rules into the catalog so detector-less
  / tool-only rules are exercisable.

Result: the native enforcer is capable of enforcing any software-engineering rule (code: forbidden token
/ pattern) and any software-architecture rule (prose/ADR: forbidden design) when a 3rd-party tool cannot
be found. Deterministic enforcement covers literal-token rules; full semantic enforcement engages under
`LLM_JUDGE=1`; everything else is surfaced as `not_enforced`, never silently dropped.

## Scope & non-regression

`--body`/`--forbid` and `--extra-rules` are additive: with none of them, `prose-judge.sh` and
`enforce-file.sh` behave byte-identically to before (all 118 catalog rules carry detectors today, so the
`universal` set is empty on the real tree). The detector-missing fallback only fires when a detector file
is genuinely absent (previously a silent skip).
