# v1.24 — §31 Phase 1 BUILT: family activation (S-58 resolver + S-59 umbrella registry)

**Built 2026-07-05.** The first, dependency-free phase of the §31 universal-technology design: naming a
technology activates its umbrella's **already-scraped** rules for the consult. No acquisition, no `_project/`
store — official surface only. Delivers the operator's headline ask ("if we see Vue in the vision, the
frontend umbrella rules should be active") without touching any boundary surface.

## What shipped

- **S-59 `standards/technology-umbrella-registry.yaml`** — `umbrellas.<name>.activates` lists the EXISTING
  framework-agnostic namespaces of a family (frontend → `owasp, web-vitals, w3c, md, typescript, node`);
  `technologies[]` maps each tech → `{coordinate{linguist,purl}, umbrellas[], specific_namespace|null}`.
  Official + PR-only (no per-project overlay — §31.4).
- **S-58 `commands/resolve-technology.sh <name>`** — resolves to coordinate + umbrella(s) +
  `activated_namespaces` (existing only) + `specific.status`:
  - `present` — the tech has its own namespace (activated too), e.g. React → `react`.
  - `needs_source` — recognized + umbrella-activated, no own namespace yet, e.g. Vue/Angular/Ember.
  - `unresolved` — not in the registry; declined, never guessed (cite-or-decline), e.g. Cobol.
  Cross-family union: a tech in several umbrellas activates the deduped union (TypeScript → frontend +
  backend-web).
- **Family activation wired into `full-surface-intake.sh --classify`** — the haystack is scanned
  (word-boundary, §30.3) for registered technologies; their activated EXISTING namespaces union into scope;
  `workload_classification.family_activated` + a `family_activated=<csv>` marker record which fired.

## The correctness fix (framework-specific ≠ category)

The workload-classifier's `web-frontend` type previously listed `react` — so *any* frontend vision inherited
React's rules. Corrected: `web-frontend` now activates only framework-agnostic namespaces; the framework-
specific `react` namespace is activated ONLY by naming React (family activation) or `--stack-add react`. So a
Vue/Angular/Ember app gets the frontend discipline (owasp/web-vitals/w3c/…) but **not** React's rules.
Verified: "dashboard built with Vue" → `react` NOT in scope; "dashboard built with React" → `react` in scope.

## Honest boundary (Phase 1)

A `needs_source` technology (Vue/Angular/Ember) gets the umbrella's existing rules now; its *own* ruleset is
acquired in Phase 2 (S-60 search-existing → `_project/`). Phase 1 fabricates nothing and touches no boundary
surface — it only activates rules that already exist.
