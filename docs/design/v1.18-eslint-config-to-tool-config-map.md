# ESLint rule-configuration → per-tool configuration map (design study)

> **Status: design study / proposal.** No substrate built yet. This studies every rule-enforcing
> tool in `rubric/runners/toolchain.json` (80 tools), maps CTP's ESLint-style per-rule
> configuration onto each tool's native rule-config model, proposes a transformation wherever it is
> not 1:1, and reports the properties that cannot be mapped (and where the whole approach is or
> isn't feasible). Exact per-tool config *keys* should be re-verified when each adapter is built;
> the **paradigm-level** mapping below is the durable design.

## 0. The linchpin that makes this feasible

CTP already (a) **resolves config centrally** (`profiles/active.sh`, threaded into enforcement in
§28.44) and (b) **normalizes every tool's output to SARIF 2.1.0** (the §28.29 bus). Therefore CTP
does **not** need each tool to natively honor severity/disable/suppression for correctness — it can
apply them **post-hoc on the SARIF stream**: map each tool finding back to the CTP rule that routed
it, then drop disabled/suppressed findings and (re)grade by the CTP rule's effective severity.
**Native-config translation (passing `--ignore`, presets, per-file-ignores, etc.) is an
optimization** — it avoids running checks that would be discarded and keeps the tool's own report
clean — **not a correctness requirement.** This single fact is why the approach is feasible across
100% of tools even where a tool has no native equivalent for an ESLint property.

## 1. Source model — ESLint's configurable properties (= CTP's config plane)

CTP's config plane is ESLint-parity (§16 E-1..E-17). The configurable properties a user sets per
rule (or per profile):

| # | ESLint / CTP property | Form |
|---|---|---|
| C1 | **Severity** | `off \| warn \| error` (`0\|1\|2`) per rule |
| C2 | **Enable / disable** | `off` (a special case of C1; the most-used knob) |
| C3 | **Rule options** | `["error", <options object / args>]` — rule-specific payload |
| C4 | **Glob overrides** | `overrides: [{ files, excludedFiles, rules }]` |
| C5 | **Extends / recommended** | `extends: [rubric:recommended, <shareable>]` |
| C6 | **Inline suppression** | `// rubric-disable[-next-line] <id> -- <reason>` (+ unused-directive report) |
| C7 | **Plugins / custom rules** | community rule packs; namespaced IDs `<plugin>/<rule>` |
| C8 | **Auto-fix** | `--fix` / `fixable: code\|whitespace\|null` |
| C9 | **Engine-local** | `env`, `globals`, `parser`, `parserOptions`/`languageOptions`, `settings` |
| C10 | **root / cascade** | `root: true`, hierarchical directory cascade |
| C11 | **Reporter / format** | `--format` (output shape, not rule config) |

## 3. Property × paradigm mapping

Legend: **=** 1:1 · **→** transformation (defined) · **✗** not mappable (carve-out).

| | A ESLint-linters | B toggle-linters | C policy-as-code | D vuln/secret | E formatters | F validators |
|---|---|---|---|---|---|---|
| **C1 severity** | = `off/warn/error` | → mostly = (clippy `allow/warn/deny`, yamllint `error/warning/disable`, vale, swiftlint, shellcheck); **collapse** where tool lacks warn≠error (ruff, golangci) → run + **CTP grades the SARIF** | → `off`=skip-check; `warn/error`= per-check severity if present, else **CTP grades** | → `off`=allowlist/ignore; `warn/error`= `--severity` **threshold** + **CTP grades** | → **C2 only**; `error` is nonsensical for style → **advisory** (formatter findings never block) | → no rule unit; **CTP grades the single verdict** |
| **C2 disable (`off`)** | = disable rule | = disable rule | = skip check | → allowlist / don't route | = don't run formatter for that concern | → don't route / ignore |
| **C3 options** | = pass native options (E-15 `detector_config.eslint_options`) | **✗ generic** — per-rule, per-tool schemas; authored **per binding** in `enforced_by[].config` | **✗ generic** — per-binding (e.g. checkov check vars) | n/a (no rule options) | n/a (style options ≠ rule options) | n/a |

## 4. The carve-outs (honest "cannot map / don't transform")

1. **C3 rule-OPTIONS payloads are not generically mappable.** ESLint rule options are per-rule,
   rule-specific schemas (`quotes:["error","single"]`, `max-len:["error",{code:100}]`). There is no
   cross-tool transformation of an arbitrary option object, because the *target* rule in another
   tool has its own, unrelated option vocabulary. This is inherent — ESLint itself cannot share an
   option schema between two of its own rules. **Disposition (not a blocker):** options live
   **per binding**, hand-authored in the rule's `enforced_by[].config` / E-15 `detector_config`,
   exactly where CTP already puts ESLint-wrap options. The *property* maps; the *payload* is
   per-(rule,tool) data, not a transformation.

2. **C9 engine-local settings do not transform and should not.** `env`, `globals`, `parser`,
   `parserOptions`/`languageOptions`, `settings` are ESLint's AST/runtime concepts. Forcing a
   Python/Go/Terraform/secrets tool to accept `env:{browser:true}` is nonsensical. **Disposition:**
   scope C9 to **ESLint-family (A) bindings only**; it is intentionally **absent** from CTP's
   cross-tool config plane. This is a "report it" item per the brief: not mappable, and transforming
   it does not make sense — so it is excluded, not faked.

3. **C7 custom-rule plugins** map only where a tool hosts custom rules. **Disposition:** route
   custom logic to the two universal custom-rule engines already in the toolchain — **semgrep**
   (code) and **conftest/regal rego** (config/policy) — else enforce CTP-native. Not 1:1, but
   covered with a documented fallback.

## 5. Feasibility verdict

**Feasible — with two clearly-bounded carve-outs; NOT a wholesale infeasibility.**

- The **governance-critical** ESLint semantics — **severity (C1), enable/disable (C2), glob
  overrides (C4), recommended/extends (C5), inline suppression (C6), fix (C8)** — map across **all
  80 tools**, via native translation where the tool supports it and via the universal **CTP-side
  resolution + post-hoc SARIF grading/filtering** fallback everywhere else. Because of the linchpin
  (§0), the fallback alone already makes C1/C2/C4/C6 correct for 100% of tools; native translation
  is a per-tool optimization layered on top.
- **Per-paradigm transformations** are well-defined and sensible: formatter `error→advisory` (E);
  vuln-scanner `severity→threshold + allowlist` (D); toggle-linter `warn/error collapse→CTP-graded`
  where the tool lacks the distinction (ruff, golangci).
- **Two carve-outs, reported as required:** (1) rule-**options payloads** are inherently
  per-binding, not transformable (C3); (2) **engine-local settings** (C9) do not transform and are
  scoped to ESLint-family only. Neither blocks the approach; both are bounded and documented.

**Conclusion:** mapping CTP's ESLint configuration onto every enforcing tool **is feasible**. The
right architecture is a **two-layer translator**: (i) a universal **post-hoc SARIF layer** (CTP
maps each tool finding → CTP rule, then applies effective severity/disable/suppression) that
guarantees correctness for all 80 tools; plus (ii) an optional, per-tool **native-config emitter**
(presets, `--ignore`, `per-file-ignores`, inline directives) that optimizes by not running
discarded checks. Rule **options** and **engine-local** settings are explicitly per-binding /
ESLint-family-scoped, not part of the cross-tool transform.

## 6. Proposed build sequence (if approved — each a gated, no-regression CL)

1. **SARIF-back-mapping** — finding → originating CTP rule (rule-id correlation table per tool).
2. **Post-hoc severity/disable** on the dispatch SARIF (C1/C2 effective on the routed path — the
   gap named in §28.44's boundary).
3. **Post-hoc inline-suppression** filter (C6) — universal, tool-independent.
4. **Override route-gating** (C4) on the dispatch path.
5. **Recommended→preset** map (C5) + **native-config emitters** (optimization) per paradigm.
6. **Per-binding options** schema (C3) wired into `enforced_by[].config`.

Each step is additive and default-preserving (no `--profile`/no config ⇒ today's behavior), gated
by the full eval suite — the same method that kept §28.44 at zero regression.
