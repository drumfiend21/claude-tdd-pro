# v1.25 — §32 Jurisdictional-namespace compliance rule authorship (BUILT)

**Built 2026-07-11.** Closes the two-gap seam surfaced by GCTP kata attempt KA-5 (P-21 in GCTP,
`docs/handoff-ctp-p21-compliance-urls-sync-gap.md`): the `.claude-tdd-pro/COMPLIANCE-URLS.yaml`
registry (C-13) declares 24 authoritative compliance frameworks with jurisdictions, the fetcher
slots exist at `compliance/fetchers/` (as stubs awaiting operator-wire logic), the target namespace
directories exist at `generated-code-quality-standards/{european-union,finance-industry,industry-self-regulatory}/`
per G-9.1 sync, and §5 line 432 mandates the mapping — but the target directories were empty, so
`active.json` carried **0 rules** for the jurisdictional namespaces even when the classifier
activated them for `regulated` workloads (KA-5 cross-check reported 2 legitimate `no-propagation`
violations flagging `european-union` and `finance-industry`).

## What this amendment authorizes

**The plugin MAY ship pre-authored rule YAMLs under
`generated-code-quality-standards/<jurisdiction-namespace>/` for the frameworks declared in
`.claude-tdd-pro/COMPLIANCE-URLS.yaml`.** Each shipped rule is grounded in a specific article,
section, or clause of the authoritative framework text, cites the framework's canonical URL from
the C-13 registry, and enforces the requirement via the same `cloud-guidance-rule.sh` prose-detector
pattern already used by `us-government/us-government-iac-enforcement.yaml` (verified pattern that
already places 2 rules into active.json).

The amendment does not require the C-14 sync mechanism or C-15 daily-fresh fetch to run first —
pre-authored rules stand alone with their `source:` header citing the C-13 registry. When
C-14/C-15 do run (future CL), fetched framework text either confirms the pre-authored rule's
correctness (green) or triggers a freshness review (rule may be edited or superseded via the
§25 fidelity gate).

## What shipped in this CL (v1.25)

- **C-24 Jurisdictional pre-authored rule pattern**: rule YAMLs under
  `generated-code-quality-standards/<jurisdiction-namespace>/<framework-id>.yaml` where
  `<jurisdiction-namespace>` matches the C-13 registry's `jurisdiction:` field
  (`european-union`, `us-government`, or `industry-self-regulatory` for `international`),
  and `<framework-id>` matches an entry's `id:` field. Each rule carries
  `provenance.section` naming the framework's article/clause and `docs_url` matching the
  C-13 registry's `url:` field.

- **C-25 Namespace-axis-binding entries** for the three jurisdictional namespaces added at
  `standards/namespace-axis-binding.yaml` — all three bind `linguist_aliases: [markdown]`
  (rules enforce prose in ADRs/architecture docs declaring the required control is in place)
  with `foss_tools: [semgrep]` polyglot fallback.

## What this closes

The GCTP-side observable effect: `active.json` regeneration after pin bump grows from 118 → 146
rules (+28 across the three jurisdictional namespaces). The GCTP `kata cross-check` invariant-4
audit on a regulated-workload architecture that activated `european-union` and `finance-industry`
probes will now find matching rules in `applicable_rules` — KA-5's 2 remaining violations resolve
to 0.

## Honest boundary

**These rules enforce PROSE compliance declarations** — they check that ADR / architecture
documents name the required control or process. They do NOT statically scan application code for
actual GDPR-compliant behavior (that's E-1 ESLint / G-13 territory for language-level rules).
The compliance rules verify that the system's DESIGN documents commit to the framework
requirement; runtime and code enforcement of the same requirements are complementary and land
under other namespaces (owasp / iam / security-governance for security-side implementation
checks, e.g. `g-security-governance-require-provenance`).

## What this does NOT do (future work)

- Does not automate rule authorship from fetched framework text (that's the future C-14 → C-24
  pipeline landing when C-15 daily-fresh fetch is wired past the stubs).
- Does not populate the remaining 15 US-government + international COMPLIANCE-URLS entries with
  pre-authored rules (follow-on CLs, one per framework family, per §20).
- Does not add a `commands/compliance-populate-jurisdictional.sh` operator command (may land in
  a follow-on CL if operator wants to author rules on their own frameworks — for now, ship-with-
  plugin is enough for the KA-5 fix).

## §25 fidelity vocabulary additions (per §25.2 append-only)

Vocabulary specs may use, valid for §32:

- `jurisdictional-namespace` — one of `european-union`, `finance-industry`,
  `industry-self-regulatory`, `us-government`.
- `pre-authored` — rule shipped with the plugin, not extracted from fetched text.
- `framework-id` — the `<framework>.yaml` filename token matching C-13 registry `id:`.
- `provenance.section` — the article/clause identifier on each rule.
- `jurisdiction-namespace-map` — the C-13 `jurisdiction:` → `<jurisdiction-namespace>/` mapping.
