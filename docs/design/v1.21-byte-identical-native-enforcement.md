# v1.21 — Byte-identical native enforcement between consult and development write-time (§29.6)

**Operator directive (2026-07-03):** "The native enforcement of all rules in the repo during
consult must be byte-identical to the same native enforcement during development which is used at
write time."

## The gap §29.5 left open

§29.5 claimed "write-time parity (byte-identical flags)" but achieved it by *duplication*: both
`hooks/scripts/enforce-standards-pre-write.sh` (development write-time governor) and
`commands/architect-session.sh` (consult) independently assembled the same
`enforce-file.sh --single-file-gate [--include-app-code]` invocation. Two copies of the flag logic
are only *identical-by-inspection* — they can silently drift (a new app-code extension added to one
caller's `case` but not the other; a flag added to one governor but not consult). "Byte-identical
flags" was a property maintained by hand, not by construction.

## The fix — one shared primitive

`rubric/enforce-write-time.sh` is now THE single write-time native-enforcement primitive. It owns the
canonical write-time flag set in exactly one place:

```bash
EA=(--file "$FILE" --single-file-gate)
case "$FILE" in
  *.ts|*.tsx|*.js|*.jsx|*.mjs|*.cjs|*.py|*.go|*.rb|*.rs|*.java|*.kt|*.php|*.cs|*.swift|*.scala|*.ex) EA+=(--include-app-code) ;;
esac
bash "$EF" "${EA[@]}"
```

Exit contract: `0` clean/advisory · `1` blocking (P0/P1) · `3` not_enforced · `2` usage.

Both callers now invoke this one script for native enforcement:

- **Development write-time** — `enforce-standards-pre-write.sh` reconstructs the proposed in-memory
  content to a scratch file and calls `bash rubric/enforce-write-time.sh "$SCRATCH_FILE"`. Exit `1`
  → deny the write (governor exit `2`).
- **Consult** — `architect-session.sh` calls `bash rubric/enforce-write-time.sh "$_af"` over every
  produced artifact and emits `design_enforcement=green|red engine=enforce-write-time
  rules_total=118`.

Because there is exactly one code path, the native enforcement of the entire repo ruleset is
byte-identical across consult and development **by construction** — not by hand-maintained
agreement. The write-time flag selection is decided in the one primitive; neither caller duplicates
it.

## Verified byte-identical

The same file yields the same native verdict from both paths. For `a.ts` = `const x: any = 1;`:

- consult-style direct call → `status=red rules_checked=22 blocking=2`, exit `1`
- development governor (Write of identical content) → embeds `status=red rules_checked=22 blocking=2`,
  denies with exit `2`

The `status= rules_checked= blocking=` summary line is character-for-character identical.

## Specs

6 specs (`cl545-byteident-01..06`):

1. both consult and development call the one shared write-time primitive
2. consult and development produce a byte-identical native verdict for the same file
3. the shared primitive is deterministic for a given file
4. the shared primitive natively enforces the fullstack rule set for app code (blocking → exit 1)
5. the write-time flag logic is not duplicated in either caller
6. a clean file is green from the shared primitive for both callers

Reconciliations (the marker/engine name moved from `enforce-file` to the shared
`enforce-write-time`, and the flags moved out of `architect-session.sh` into the primitive):
`cl543-abide-08`, `cl544-parity-01`, `cl544-parity-03`.
