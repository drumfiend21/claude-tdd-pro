# Release engineering

Per the simulated Jez Humble review note, the project's CI/CD
maturity stops at "merged to main." This document defines the
release process to close the gap to true CD.

## Versioning

[Semantic Versioning 2.0.0](https://semver.org/) from 0.4.0 onward.

- **MAJOR (X.0.0)** — breaking change to the architecture (a §X is
  removed; a feature ID is renamed; the rubric rule schema §2.1 is
  altered). Requires a governance CL with explicit MAJOR justification.
- **MINOR (0.X.0)** — backwards-compatible additions: a new feature
  ID in an existing §, a new fitness function, a new amendment block
  (§26, §27, etc.).
- **PATCH (0.0.X)** — fixes, polish, doc-only changes,
  non-architectural substrate refinements.

## Release cadence

- **PATCH releases**: continuous as fixes ship.
- **MINOR releases**: per architecture amendment or quarterly,
  whichever comes first.
- **MAJOR releases**: by intent, with at least one prior MINOR
  release that previews the change behind a feature flag.

## Release procedure

### 1. Confirm release-readiness

```bash
# Suite green
npm test

# All fitness functions clean
npm run drift:audit

# Lockfile current vs main HEAD
bash scripts/install.sh upgrade --yes
```

### 2. Update CHANGELOG

Move the `[Unreleased]` block to a versioned block in `CHANGELOG.md`:

```markdown
## [0.4.1] — 2026-06-04

### Added
- ...
```

### 3. Bump version

```bash
# In package.json
sed -i.bak 's/"version": "[^"]*"/"version": "0.4.1"/' package.json

# In scripts/install.sh
sed -i.bak 's/VERSION="[^"]*"/VERSION="0.4.1"/' scripts/install.sh

# In README.md
sed -i.bak 's/version-[0-9.]*-/version-0.4.1-/' README.md
sed -i.bak 's/Current version: [0-9.]*/Current version: 0.4.1/' README.md

rm -f *.bak
```

### 4. Verify version coherence

```bash
grep -E "VERSION=|version-|Current version" \
  package.json scripts/install.sh README.md CHANGELOG.md
```

Every output line should show the same version string.

### 5. Tag and commit

```bash
git add CHANGELOG.md package.json scripts/install.sh README.md
git commit -m "release: 0.4.1"
git tag -a v0.4.1 -m "Release 0.4.1 — see CHANGELOG.md"
git push origin main
git push origin v0.4.1
```

### 6. Cut a GitHub release

```bash
# (requires gh CLI; see .github/workflows/release.yml for the
#  automated path)
gh release create v0.4.1 \
  --title "v0.4.1" \
  --notes-file <(awk '/^## \[0.4.1\]/,/^## \[/' CHANGELOG.md | head -n -1)
```

The release artifact is a source tarball auto-generated by GitHub
from the tag.

### 7. (Optional) Publish to npm

Once the package surface stabilizes:

```bash
npm publish --access public
```

This makes the installer available as:

```bash
npx claude-tdd-pro init
```

## Canary releases

The installer's lockfile model (`.claude-tdd-pro.lock.json`) gives us
canary capability for free:

- A "canary" release is tagged on a feature branch
  (`canary-0.5.0-rc.1`) and consumed by passing
  `install.sh init --pin canary-0.5.0-rc.1`.
- Consumers opt-in to the canary; default install always pulls
  `main` HEAD (which equals the latest stable tag at any moment
  after a release).
- Canary feedback is collected in GitHub issues tagged `canary`.
- After ≥7 days of clean canary, the canary tag is promoted to a
  full release tag.

## Rollback

If a release ships with a regression that fitness functions failed to
catch:

1. **Immediate**: consumers re-pin to the previous tag:
   `install.sh init --pin v0.4.0 --force`.
2. **Repo state**: revert the merge commit on `main`; cut a `0.4.2`
   release containing only the revert.
3. **Postmortem**: file an issue tagged `regression` and `fitness-gap`
   documenting which fitness function should have caught the issue,
   add it to the roadmap in `docs/FITNESS_FUNCTIONS.md`.

## Why this matters

Per Humble: "Without this, you're at CI-mature with manual deploys."
The release pipeline is what separates "we ship sometimes" from
"we ship continuously and recoverably." The procedure above
documents the discipline; the workflow at
`.github/workflows/release.yml` automates the mechanics.
