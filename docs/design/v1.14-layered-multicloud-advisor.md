# v1.14 — Layered Multi-Cloud Advisor: Common Business-to-Technical Core + Three Platform Boundaries + Data/Distributed Domain

**Status:** designed (2026-06-08). Registered additively as architecture §27.16 (features S-37, S-38, S-39, S-41, S-42, S-43, S-44, S-45; plus the option-composer design for the already-registered S-34).
**Goal:** abstract the cloud platforms behind one **common, platform-agnostic business-to-technical layer** that holds all cross-platform best-practice patterns (cloud + data + distributed-systems/integration) and guides beginners in business language; then, in **three platform-expert boundaries (AWS / Azure / GCP)**, add platform-specific knowledge and wrap each vendor's native advisory API.

## 1. Layered architecture

```
            BEGINNER / BUSINESS USER  (plain business language, via S-36)
                                |
   COMMON BUSINESS-TO-TECHNICAL LAYER  (platform-agnostic, fully grounded)
     S-32 intake · S-33 translate · S-34 multi-option composer · S-35 explain
     Cross-platform pattern knowledge: cloud (AWS/Azure/GCP WAF + NIST/DoD) +
     data-architecture (S-37) + distributed-systems/integration (S-37)
                                |
        ----------------------- S-41 platform-boundary contract -----------------------
        |                              |                              |
   AWS BOUNDARY (S-42)          AZURE BOUNDARY (S-43)          GCP BOUNDARY (S-44)
   AWS pattern depth +          Azure pattern depth +          GCP pattern depth +
   WA Tool API wrapper          Advisor API wrapper            Recommender API wrapper
   (injected/normalized)        (injected/normalized)          (injected/normalized)
                                |
          TDD BUILD (S-29, platform-aware) + ENFORCE (S-30) + TOOLCHAIN ADVISOR (S-45)
```

The beginner only ever talks to the **common layer**. A platform boundary is invoked **only** once an option is selected (or when S-34 proposes a platform-specific option). Boundaries add native depth without polluting the common knowledge.

## 2. Common layer — platform-agnostic core (the beginner's whole experience)

Extends the v1.13 layer (S-32 intake, S-33 translate, S-34 recommend, S-35 explain) with two things:

- **Cross-platform pattern knowledge (S-37 sources).** Two new grounded catalogs:
  - `standards/data-architecture-sources.yaml` — [AWS Data Analytics Lens](https://docs.aws.amazon.com/wellarchitected/latest/analytics-lens/analytics-lens.html), Azure Data Architecture Guide, Google Cloud data-architecture guidance, plus the consistency/replication/partitioning pattern vocabulary. (Kleppmann's *Designing Data-Intensive Applications* is referenced as a book; we register the free primary docs, not its text.)
  - `standards/distributed-systems-sources.yaml` — [Enterprise Integration Patterns free catalog](https://www.enterpriseintegrationpatterns.com/patterns/messaging/) (queues/messaging/server-to-server, internal↔external integration), [Patterns of Distributed Systems (martinfowler.com)](https://martinfowler.com/articles/patterns-of-distributed-systems/) (consensus, replication, partitioning), and Martin Fowler's CQRS / Event Sourcing / Saga pages.
- **S-34 as the multi-option composer (the "creative flow").** See §4.

### S-38 — data-aware intake (extends S-32)
Adds business-language questions, each grounded: data volume & growth, read/write pattern (OLTP vs analytics), consistency need (strong vs eventual), communication style (synchronous vs event-driven), integration scope (internal-only / external-partner / public), and cadence (real-time vs batch). Still enum-validated, still cite-or-decline.

### S-39 — data & distributed translation (extends S-33)
New mapping rules → pillar/domain concerns, grounded in the S-37 sources, e.g.:
- high write throughput + strong consistency → `sharding`, `synchronous_replication`, `partition_strategy` (data-architecture sources)
- event-driven communication needed → `message_queue`, `dead_letter_queue`, `idempotent_consumer`, `outbox_pattern` (Enterprise Integration Patterns)
- cross-service workflow + failure isolation → `saga`, `cqrs`, `event_sourcing` (Fowler / Patterns of Distributed Systems)
- internal↔external integration → `api_gateway`, `anti_corruption_layer`, `contract_test`

## 3. Three platform boundaries (expert layers) — S-41/42/43/44

Each boundary contains (a) platform-specific pattern/service knowledge and (b) a thin wrapper around the vendor's native advisory API. Determinism is preserved exactly as elsewhere in the plugin: **the native API response is an injected data source** (like S-21's injected HTTP and S-24's injected deltas); the deterministic substrate is the *normalizer* that maps native findings into the common grounded concern/option format. cite-or-decline still applies — a native finding that can't be mapped to a grounded concern is marked `needs_grounding`.

- **S-41 Platform-boundary contract & dispatcher.** Defines the handoff envelope and routes a selected option to the right boundary. Handoff in: `{business_profile, technical_requirements, selected_option, target_platform}`. Handoff out: `{platform, recommendations[], iac_targets[], build_units[], native_review_ref}`.
- **S-42 AWS boundary.** AWS service/pattern depth; wraps the [AWS Well-Architected Tool API](https://docs.aws.amazon.com/wellarchitected/latest/APIReference/Welcome.html) (CreateWorkload / AssociateLenses / UpdateAnswer / GetLensReview, including the Data Analytics Lens). Normalizes risks/best-practices into common concerns; platform-aware IaC = Terraform/CloudFormation.
- **S-43 Azure boundary.** Azure service/pattern depth; wraps the Azure Advisor REST API (recommendations by WAF pillar). Platform-aware IaC = Bicep.
- **S-44 GCP boundary.** GCP service/pattern depth; wraps the [GCP Recommender API](https://docs.cloud.google.com/recommender/docs/reference/rest) (personalized cost/perf/security/reliability recommendations). Platform-aware IaC = Terraform/Deployment Manager.

Each boundary's API call is the thin I/O edge; in tests it is a fixture, so the suite stays hermetic.

## 4. S-34 — the multi-option composer (copy & improve on AWS Well-Architected Tool)

**Closest product:** the AWS Well-Architected Tool (+ Profiles + Lenses) — it asks technical questions and returns prioritized findings + improvement actions. **We improve on it** by:
1. starting from **business language** (S-32/S-33), not technical questions;
2. emitting **2–4 creative but grounded options** with explicit **trade-offs** (cost, complexity, latency, consistency, operational burden, vendor-lock-in), not a single findings list;
3. **multi-authority grounding** (AWS + Azure + GCP + NIST/DoD + EIP + data/distributed patterns) with cite-or-decline;
4. each option feeds **S-28 ADR** + **S-29 TDD build** so a choice becomes an enforceable artifact;
5. options can be cross-cloud ("one AWS-native, one Azure-native, one portable").

S-34 output per option: `{option_id, summary, patterns[], pillar_coverage, trade_offs{...}, drivers[], grounding[], suggested_adr_title, build_requirements[]}`. The native boundary APIs (S-42/43/44) **enrich** these options; they are not the final answer.

## 5. S-45 — implementation-toolchain advisor (beyond IaC)

Terraform/Bicep only provision infrastructure. S-45 recommends and (where deterministic) scaffolds the rest of the toolchain for a chosen option + platform, each recommendation grounded:

| Category | Representative tools the advisor maps to |
|---|---|
| IaC / provisioning | Terraform, Bicep, CloudFormation, Pulumi, Crossplane |
| Containers / orchestration | Docker, Kubernetes (EKS/AKS/GKE), Helm, Kustomize |
| GitOps / CD | Argo CD, Flux (already a grounded source) |
| CI/CD | GitHub Actions, GitLab CI, CodePipeline, Cloud Build |
| Config management | Ansible |
| DB migrations | Flyway, Liquibase, Alembic |
| Messaging / streaming | Kafka/MSK, SQS/SNS, Service Bus, Pub/Sub, RabbitMQ |
| Observability | OpenTelemetry, Prometheus, Grafana, Loki (OTel already grounded) |
| Policy-as-code / security | OPA/Gatekeeper, Kyverno, Checkov, tfsec, Trivy, Vault |
| Testing | Terratest, Pact, chaos (FIS/LitmusChaos) |
| FinOps | Kubecost, native cost tools (FinOps already grounded) |

S-45 emits a grounded `toolchain.json` per option, and (for the deterministic parts) scaffolds starter configs as S-29 build units (e.g., an Argo CD Application manifest, an OPA policy, a Flyway migration skeleton) — each with a conformance check.

## 7. Non-goals / honest constraints
- No live cloud credentials in the hermetic suite — native API responses are injected fixtures; the deterministic substrate is the normalizer.
- No reproduction of book text (EIP, DDIA, Patterns of Distributed Systems) — we register free primary catalogs/URLs and ground rules in them.
- Boundaries enrich; the common grounded layer remains the source of truth. cite-or-decline everywhere.
