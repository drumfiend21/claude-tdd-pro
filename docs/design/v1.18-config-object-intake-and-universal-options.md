# v1.18 — Config-object intake + universal options projection (2026-06-30)

Operator directive (three coupled requirements):
1. *"New rules, when added, need a config object added to the single config file, configurable for
   whatever tool it's mapped to. There has to be an intake process triggered anytime the source URLs
   change / new rule content is scraped."*
2. *"Extend the generic universal config object so the gap tools' (checkov/semgrep) options can be
   provided in the single config for any given rule."*
3. *"Capability present, data empty — nothing can be missing. Everything wired from the architecture."*

This closes the option-projection gap (2 of 9 tools, 51 rules) AND the options-data gap (0 of 118 rules
populated). **No new feature ID, no new §2.X contract** — it composes ADR-0009 (classify → route →
draft), the §28.50 option-surfaces, the §28.51 emitter, and the §28.48 single config.

## Part A — universal options projection (the generic config object)

The per-rule `options` container is already generic (`schemas/rubric-rule.schema.json`
`enforced_by[].options` is `additionalProperties: true`). What was missing was projection for two
CLI-configured tools and one mis-flagged tool:

- **`emit-tool-config.sh` second render METHOD: `cli`.** A tool configured by CLI flags declares
  `render: { method: cli, map: { <option-key>: <flag> } }`. Options project to a flag string (bool true
  → bare flag; scalar → `flag value`; array → flag repeated; an **unmapped** key → generic
  `--<key> <value>`, so *any* option for *any* tool is providable). Emitted to `<tool>.flags` and printed
  to stdout for the runner to splice. The existing file methods (json/yaml/toml/ini → config file +
  `--flag <path>`) are unchanged.
- **`run-tool.sh` injection.** Generic path: `method: cli` → splice the flag string; file method →
  inject `<flag> <config-path>` (unchanged). The bespoke **checkov** adapter now emits `.checkov.yaml`
  and injects `--config-file`.
- **`tool-option-surfaces.yaml` gap tools fixed:** checkov → `render {fmt: yaml, file: .checkov.yaml,
  flag: --config-file}` (was `supported:false`); semgrep → `render {method: cli, map: {severity,
  exclude, timeout, max_target_bytes}}`; trivy → `render {method: cli, map: {severity, ignore_unfixed,
  exit_code}}`. **All 9 referenced 3rd-party tools now project.**

## Part B — config-object intake / materialization (ADR-0009 stage 6)

`commands/config-sync.sh` materializes the options DATA so the single config is never empty:

- For each rule, reads `enforced_by[]`; for every 3rd-party tool, **seeds** a `options.<tool>` object
  from the tool's documented vocabulary (`tool-option-surfaces.yaml` `examples`), producing
  `<rule-id>: { severity, options: { <tool>: { ... } } }`.
- `--rule-id <id> --enforced-by <csv>` materializes one rule; `--all` materializes the whole catalog as
  the single-config `rules:` map; `--check` exits 1 if any active rule has a 3rd-party tool with no
  projectable option surface (the **nothing-missing gate** — currently `needs_mapping=0` over all 118).
- **cite-or-decline:** a tool with no projectable surface is recorded `needs_mapping` and surfaced,
  never silently dropped.

## The trigger (intake on source change)

`config-sync` is **ADR-0009 stage 6**, the tail of the existing intake pipeline:

```
standards-monitor.sh (S-10, source delta via S-21 conditional-GET 304/changed)
  → extract → classify-rule.sh (4-axis) → route-rule.sh (enforced_by[])
  → draft-custom-rule.sh (tool DSL) → config-sync.sh (materialize options object) → ctp.config.yaml
```

It fires whenever the source URLs change and new rule content is scraped (S-20 poll cadence; S-21
304 → cache hit / no-op). A CI `config-sync --check` gate fails the build if any active rule lacks a
projectable options object — so "data empty" can never re-enter. `--all` is also run on `standards-add`
(S-14) and `promote-standard` (S-7) when a new rule lands.
