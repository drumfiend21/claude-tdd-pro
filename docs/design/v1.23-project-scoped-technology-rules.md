# v1.23 — Project-scoped technology rules: activate-existing + search-existing + per-project store + PR-promote (§31.1)

**Operator correction (2026-07-05) to §31.** Two things in the v1.22 design were wrong; this supersedes them:

1. **Umbrella rules already exist.** The sources the operator provided (Google front-end engineering, OWASP,
   web-vitals, WCAG, …) are already scraped into the surface. Recognizing Vue/Angular/Ember → *frontend* must
   simply **activate those existing umbrella rules** — not define new "general namespaces."
2. **Dynamic sourcing = search the SAME existing sources for the specific technology** — NOT discover a new
   authoritative source. Re-query the URLs already in `standards/*sources*.yaml` for content specific to
   Vue/Angular/Ember, extract a *new tech-specific ruleset* (as React has one today), tag it, and store it.

Plus three additions the operator specified: a **per-project rule folder**, **PR + code-review promotion** to
the plugin core, and **first-class enforcement from stage zero** the moment the rules are tagged.

**Status:** DESIGN. Refines §31's S-60/S-62; adds S-63/S-64; contract §2.38.

## 0. The corrected flow (Vue example)

```
vision cites "Vue"
   │
   ├─(A) resolve Vue → frontend umbrella → ACTIVATE the EXISTING frontend rules
   │        (google-*, owasp, web-vitals, wcag …) for consult + design + development     [already scraped]
   │
   └─(B) SEARCH the existing source corpus (the same URLs already provided) for Vue-specific guidance
            → extract Vue-specific rules (like the React ones)
            → tag 4-axis (linguist:Vue · purl:pkg:npm/vue)  [as usual]
            → store in the PER-PROJECT folder  _project/<project-id>/vue/*.yaml  (origin: "project")
            → aggregator picks it up → ENFORCED from stage zero → architecture → development → write-time
            → OPTIONAL: promote into the plugin core (generated-code-quality-standards/vue/) via PR + review
```

(A) always works (rules exist). (B) yields tech-specific rules **iff** the existing sources contain
Vue-specific guidance; if they don't, only the umbrella rules apply — honestly, with no fabrication.

## 1. Reuse the aggregator's existing origin machinery — do not invent a new one

`rubric/aggregator.sh` already walks and origin-tags four rule categories under
`generated-code-quality-standards/`:

| Folder | `origin` | Meaning |
|---|---|---|
| `_universal/` | (baseline) | cross-cutting rules |
| `<namespace>/` (google, aws, react…) | `plugin` | core plugin rules |
| `_operator/` | `operator` | operator-added rules |
| `_community/<plugin-id>/<ns>/` | `community` | community-plugin rules |
| **`_project/<project-id>/<ns>/`** (NEW) | **`project`** | **per-project acquired rules** |

Per-project rules are simply a **fifth origin category**, walked exactly like `_community/<plugin-id>/`. That
makes them **first-class in the surface** (the aggregator emits them → the enforcer, classifier, consult and
write-time all see them) while remaining clearly **origin-tagged and project-scoped**, so they never silently
pollute the committed core and can be filtered by origin.

## 2. Feature decomposition (refines §31)

- **S-58 / S-59 (refined) — activate EXISTING umbrella rules.** The resolver maps a framework
  (Vue/Angular/Ember) to its umbrella and activates the umbrella's **already-present** namespaces (frontend →
  the existing `google-*`/`owasp`/`web-vitals`/`wcag` rules). No general-namespace definition; the umbrella
  registry just names which existing namespaces an umbrella activates.
- **S-60 (refined) — search-existing-sources acquisition.** `commands/acquire-technology-rules.sh <coordinate>`
  re-runs the **existing fetchers** against the **existing source URLs** (`standards/*sources*.yaml`), filtered
  for the technology, extracts tech-specific rules, tags them 4-axis (`namespace-axis-binding`), attaches
  provenance (`source`/`url`/`fetched_at`/`tier`), and writes them to `_project/<project-id>/<ns>/`. No
  new-source discovery; no invented rules. If the corpus yields nothing tech-specific → only umbrella rules
  apply (§2.36 `needs_source` stays, honestly).
- **S-63 (NEW) — per-project rule store.** `generated-code-quality-standards/_project/<project-id>/<ns>/*.yaml`,
  `origin: "project"`, walked by the aggregator like `_community`. Gitignored in the plugin (project-local,
  machine-local) so acquisition never dirties the committed core. Contract §2.38.
- **S-64 (NEW) — promote to core via PR + code review.** `commands/promote-project-rule.sh <project-id> <ns>
  <rule-id>`: opens a **pull request** moving the rule from `_project/<id>/<ns>/` into the core `<ns>/` tree
  (origin `plugin`), so it becomes a permanent plugin rule only after human code review. The governance gate
  from project-local → plugin-permanent.
- **S-62 (refined) — first-class from stage zero.** The instant a rule lands in `_project/` and is aggregated,
  it is enforced identically to core rules: **stage-zero detection** (classify sees the namespace),
  **consultation** (§30 probes + grounding), **architectural design** (§29.4/§29.6), and **development
  write-time** (§29.6 byte-identical; the 4-axis tag routes it to its file kind + `foss_tools`). "As soon as
  tagged, enforced everywhere" is the invariant.

## 3. Contract

- **§2.38 Per-project rule store + PR promotion.** Acquired tech-specific rules live under
  `_project/<project-id>/` with `origin: "project"`; they are **first-class in the aggregator/enforcer surface
  for that project** (enforced stage-zero → write-time, scanned + consulted like every other rule) the moment
  they are tagged; they are **gitignored** in the plugin until promoted; and they enter the plugin core **only
  via pull request + code review** (S-64). Every project rule carries a 4-axis tag + provenance (composes
  §2.37); no rule is hand-authored or inferred; the §2.6 freshness/tier gates apply.

## 4. Two-layer honesty (restated correctly)

- **Umbrella layer** — always applies, because the rules already exist. Vue/Angular/Ember get the full
  existing frontend discipline (Google/OWASP/web-vitals/WCAG) with zero acquisition.
- **Tech-specific layer** — acquired by searching the *existing* sources for the technology. Present iff those
  sources actually contain tech-specific guidance. When present → stored per-project, enforced first-class,
  promotable to core via PR. When absent → only the umbrella layer applies; nothing is fabricated.

## 4a. The acceptance invariant — the official ruleset is PR-gated (hard)

Two directories, one gate:

| Directory | Role | `origin` | Enforced? | Official? |
|---|---|---|---|---|
| `_project/<project-id>/<ns>/` | **new working rules** (acquired, candidate) | `project` | ✅ first-class, for that project | ❌ |
| `generated-code-quality-standards/<ns>/` | **official rules** | `plugin` | ✅ everywhere | ✅ |

**INVARIANT: a rule becomes OFFICIAL only through an approved pull request that MOVES it from the working
directory (`_project/…`) to the official directory (`<ns>/…`). There is no other path in.** Acquisition
(S-60) only ever writes to `_project/`; it can never write to an official namespace. `promote-project-rule`
(S-64) does not *accept* the rule — it *opens the PR that requests the move*; acceptance is the human code
review + merge. So:

- **Working rules are used, not accepted.** They enforce for their project immediately (first-class from stage
  zero), but they are never part of the official corpus and never ship in the plugin until reviewed.
- **The official directory changes only via reviewed PR.** No command, acquisition, or automation writes into
  an official namespace. This preserves the existing constitutional guarantee that the official ruleset is
  curated and reviewed — the working store is the *only* place automation may write.
- **The move is a promotion, not a copy.** A merged promotion PR removes the rule from `_project/` (it is now
  official under `<ns>/`, origin `plugin`); an un-merged/closed PR leaves it working-only.

## 5. Compatibility + rollout

Fully additive. `_project/` is a new origin category alongside the existing `_operator`/`_community`; the core
44 namespaces, v1.0/v1.1 profiles, and every existing rule are untouched. Per-project rules are gitignored, so
acquisition can never dirty the plugin core; promotion is an explicit, reviewed PR. Sequencing (§20): S-58/S-59
umbrella activation → S-63 per-project store → S-60 search-existing acquisition → S-62 first-class enforcement
→ S-64 PR promotion.

## 6. Acceptance sketch

- Umbrella activation: "Vue" in the vision activates the existing frontend namespaces for consult + design +
  write-time; no vue namespace required.
- Search-existing acquisition: acquiring Vue rules re-queries only the existing source URLs; extracted rules
  carry provenance to those sources; 4-axis tagged; land in `_project/<id>/vue/`.
- Per-project store: the aggregator emits `_project/<id>/vue/` rules with `origin: "project"`; they appear in
  the surface for that project and are gitignored from the core.
- First-class enforcement: a `_project/` rule is enforced at write-time on its file kind byte-identically
  (composes §29.6) and is visible at stage-zero classify.
- PR promotion: `promote-project-rule` opens a PR moving the rule into the core `<ns>/`; core is unchanged
  until the PR merges.
- Honesty: a technology whose existing-source search yields nothing gets umbrella rules only; no fabricated
  tech-specific rule appears.
