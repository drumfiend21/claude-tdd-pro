# v1.18 — Govern-before-write: enforce proposed content in memory before it is saved (2026-06-30)

Operator directive: *"Have the enforcement done before writing to file, governing the generation of
content to memory."*

## The shift

§28.27 write-time enforcement was **after** the write (`enforce-standards-on-save.sh`, a PostToolUse
hook): the file is written, then audited, and a P0/P1 violation surfaces *after* the bytes hit disk.
This adds a **PreToolUse** governor that evaluates the PROPOSED content **in memory, before the write**,
so a violating file is never persisted in the first place.

## The mechanism

`hooks/scripts/enforce-standards-pre-write.sh` (PreToolUse, matcher `Edit|Write|MultiEdit`):

1. Reconstructs the proposed content from the tool input:
   - **Write** → `tool_input.content` (the full proposed file);
   - **Edit** → the current file with `old_string`→`new_string` applied (honoring `replace_all`);
   - **MultiEdit** → the current file with every edit applied in sequence.
2. Writes the proposed content to an in-memory scratch file with the target's basename (so the §2.1
   `applies` globs match), and runs `rubric/enforce-file.sh` against it.
3. A **P0/P1 blocking violation → exit 2**, which DENIES the tool call and surfaces the violations to the
   model — the file is never written. Clean / advisory-only content → exit 0 (allowed). The governor
   does NOT write the target itself; it only evaluates, so on a denial the on-disk file is unchanged.

It governs the same config/markup/IaC kinds as the post-write hook (`*.yaml/json/md/tf/bicep/...`);
JS/Py stay with `lint-on-save`. Fail-open like the other write hooks: unparseable input / missing dep /
defense-trip → exit 0 (never a spurious block).

## Relationship to the after-write phase

The PostToolUse `enforce-standards-on-save.sh` REMAINS as the after-write backstop (it additionally runs
the routed FOSS tools, the architectural-content bundle, and the resolved profile). So enforcement now
has two moments on the same engine: **govern-before-write** (deterministic native gate, blocks the write)
and **audit-after-write** (full composite engine + profile). Together with §28.56–§28.59 the native gate
can enforce any applicable rule on any file type, so content is governed as it is generated to memory.
