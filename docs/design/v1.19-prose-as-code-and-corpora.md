# v1.19 — Prose-as-code detector + YAML/JSON/MD rule corpora (design)

Detailed design backing architecture amendment **§28.24** and
[ADR-0007](../adr/0007-yaml-json-md-corpora-and-prose-judge.md). Lands
PROPOSAL-003 (the CTP work brief). The architecture file carries only the
extractable reference block; the rationale, mapping tables, detector
contracts, and the three-wave ticket plan live here.

## 1. Why this work exists

The consuming harness (GCTP) ran a real engagement and found the dead zone:
CTP enforces **code shape** thoroughly (TypeScript, React, IaC, polyglot
universal patterns) but the **architectural prose** that *decides* the design —
the ADR proposing `0.0.0.0/0` ingress, the design doc choosing `any`,
the README promising a secret in plaintext — was never enforced against the
same rules. The implementing code gets caught; the decision that caused it
does not. PROPOSAL-003's empirical finding: the same `g-aws-no-unrestricted-ingress`
rule that fires on a `.tf` should fire on an ADR that *proposes* unrestricted
ingress, **before** the Terraform is written.

This is the **prose-as-code principle**: one rule, one gate, two surfaces
(code-shape detector + prose projection).

## 2. The architecturally novel piece (the only genuinely new substrate)

Per the brief's §11, of everything proposed the single novel mechanism is
**CTP-D-2 (`applies_to_prose` flag) + CTP-D-3 (`prose-judge.sh`)**. Everything
else — 22 namespaces, Layer-1 tool wrappers, SARIF, refresh entries — is
**rule-content density**, which CTP already knows how to produce. So this CL
lands the substrate in full and a Wave-1 down payment of rule content; the
remaining rule density follows wave-by-wave (each its own pin bump).

### 2.1 `applies_to_prose` (CTP-D-2) — additive rule-shape extension

Two optional fields added to `schemas/rubric-rule.schema.json`
(`additionalProperties:false`, so they had to be declared):

- `applies_to_prose: boolean` — default `false`. When `true`, the rule also
  binds architectural Markdown prose via `prose-judge.sh`, not only its
  code-shape detector.
- `applies_to_prose_kinds: string[]` — default `["architecture","adr"]`. Which
  Markdown section kinds prose-judge evaluates the rule against.

Default-`false` is deliberate (brief §5): defaulting every rule to prose
enforcement is too aggressive — many syntactic rules have no meaningful prose
analog and would generate ABSTAIN noise. Promote per rule.

The flag lives **on the rule shape**, not in an internal feature flag (brief §5
rejected alternative), so the channel is transparent to the rule registry and a
consuming harness's static gates can enforce the prose floor.

### 2.2 `prose-judge.sh` (CTP-D-3) — the semantic-projection detector

`rubric/detectors/prose-judge.sh --rule <id> [--paths <glob>] [--root <dir>] [--json] [--llm-judge]`

Generalizes the proven `LLM_JUDGE=1` shell-out (already in `no-any.sh` /
`naked-throw.sh` → `llm-judge.sh`) into a first-class detector: take any rule
body + any prose section → YES (violates) / NO (compatible) / ABSTAIN.

Three tiers, cheapest first:

1. **Keyword tier (deterministic).** If the rule forbids a literal token (read
   from `cloud-guidance-rules.json` / `universal-pattern-rules.json` manifests,
   `mode: forbid`) and the prose contains it → `violates`. Catches an ADR that
   says "leave ingress unrestricted (0.0.0.0/0)". No model, no cost.
2. **LLM tier (`LLM_JUDGE=1` + `llm-judge.sh` + a model CLI).** Semantic
   YES/NO/ABSTAIN on paraphrased intent. Best-effort, non-fatal.
3. **Fallback.** A section that mentions the rule's keywords but is unjudgeable
   (no model available) → `not_enforced` — RED, **never a silent green**.
   Otherwise `compatible`.

- **Caching:** by `sha256(rule_body + literals) + sha256(section)` in
  `<root>/.claude-tdd-pro/cache/prose-judge/`. Re-runs on unchanged prose cost
  ~0. Eager mode is opt-in only (`CLAUDE_TDD_PRO_PROSE_JUDGE_EAGER=1`); never
  runs on every MD every session (brief §5 rejected eager-by-default).
- **Sectioning:** the `.md` is split on ATX headings (`#`..`######`); each
  section judged independently so the SARIF location points at the offending
  heading's line.
- **Output:** SARIF 2.1.0 (`--json`). Exit `0` green / `1` red (≥1 violates) /
  `3` incomplete (not_enforced, no violates) / `2` usage. The 4-state verdict
  mirrors the §28.17 `enforce.sh` contract so a not_enforced prose check never
  collapses to success.

## 3. The corpora (rule-content density — staged by wave)

CTP-D-1 adds 22 namespaces; CTP-D-4 adds Layer-1 tool wrappers; CTP-D-5 makes
SARIF the universal detector output; CTP-D-6 extends refresh; CTP-D-7 bumps
the `active.json` schema_version. None of these are novel substrate — they are
rule content authored against the §6 source manifest
([docs/standards-source-manifest.md](../standards-source-manifest.md)).

### 3.1 The 22 namespaces (CTP-D-1)

`yaml · k8s · helm · compose · gha · glci · azdo · circleci · bbp · jenkins ·
ansible · cfn · oas · gitops · observability · mesh · iac-linter · json ·
jsonschema · iam · sbom · sarif · jwt · md · arch`

Each materializes as `generated-code-quality-standards/<ns>/<file>.yaml` with
§2.1 rules (`g-<ns>-*` ids), a detector under `rubric/detectors/`, grounded
`provenance[]` pointing at the source manifest, registered in
`validate-all.sh KNOWN_NAMESPACES`, and an `enforce.sh` ns_glob mapping.

### 3.2 Wave plan (each wave = one pin bump; §20 sequencing)

| Wave | Scope | ~rules | Unlocks |
|---|---|---:|---|
| **1** | `yaml` + `k8s` + `md` (Layer 1) + `arch` (template-shape) | ~50 | Closes the MD syntactic + ADR-shape dead zone. No new LLM cost. |
| **2** | `json` + `jwt` + `iam` + `sbom` + `sarif` | ~60 | P0 cluster (JWT BCP RFC 8725 + IAM wildcards); SchemaStore; SARIF self-conformance. |
| **3** | all CI/CD/IaC (`gha`/`glci`/`azdo`/`circleci`/`bbp`/`jenkins`/`ansible`/`cfn`/`oas`/`gitops`/`observability`/`mesh`/`iac-linter`) + `prose-judge.sh` LLM engine activation | ~100+ | Full corpus; prose-as-code activated — `g-aws-*` fires on ADR prose. |

**This CL's down payment (Wave-1 seed):** the `md` namespace
(`g-md-fenced-code-language-declared`, `g-md-single-h1`) via the new
`md-structure.sh` deterministic Layer-1 detector, fully wired through
`enforce.sh` (4-state). The substrate (`prose-judge.sh` + `applies_to_prose`)
ships complete now so Wave 3's LLM activation needs no further substrate work.

### 3.3 Layer-1 detector wrappers (CTP-D-4) — staged

`md-syntax.sh` (markdownlint), `yaml-syntax.sh` (yamllint), `json-schema.sh`
(Ajv + SchemaStore), `md-prose.sh` (Vale), `md-links.sh` (lychee),
`md-spell.sh` (cspell+codespell), `mermaid.sh`, `yaml-iac.sh`, `json-iam.sh`,
`sbom-validate.sh`, `jwt-bcp.sh`, per-platform CI detectors. Each emits SARIF
2.1.0; each returns the 4-state verdict (a missing tool → `not_enforced`, never
a vacuous green). `md-structure.sh` (this CL) is the deterministic, dependency-
free member of the family — it needs no external binary, so it is the safe
Wave-1 seed.

## 4. License posture (CTP-D-6)

- **Mirror** (store the rule body derived from the source): Apache-2.0, MIT,
  BSD, CC-BY / CC-BY-SA, CC0, MPL, ISC, public-domain (NIST/US-gov).
- **Config-only mirror** (store only derived config, not prose): GPL tools
  (yamllint, ansible-lint, codespell).
- **Cite-link only** (no mirror; `provenance[].url` + a derived rule we author
  ourselves): AWS, Microsoft/Azure, CIS, Snyk, Atlassian, ISO (paywalled →
  use arc42 + C4 surrogates).

`standards/initial-refresh.sh` enumerates the manifest sources; each rule's
`provenance[]` carries the source id + section, so the §2.33 citation
auditor stays green and the §28.22/§28.23 refresh loop tracks upstream.

## 5. Boundary discipline (brief §8)

CTP owns rule corpora + detectors + the `prose-judge` engine. The consuming
harness (GCTP) consumes CTP as a **pinned plugin**. The only contract surface
that moves between them is `active.json` (the rule catalog) +
`rubric/detectors/` (the executable contract). **CTP does not edit GCTP and
must not know about its consumers; GCTP does not edit CTP.** Each wave is
adopted consumer-side via the harness's own pin-bump ADR.

## 6. Alternatives considered (recorded in ADR-0007 §Considered options)

1. Single mega-detector for all 22 namespaces — REJECTED (violates §2.2 per-rule
   detector contract; opaque failure attribution; precludes per-rule SARIF).
2. prose-judge as internal feature flag, not on the rule shape — REJECTED
   (hides opt-in from the registry; consumer static gates cannot enforce it).
3. Default `applies_to_prose: true` — REJECTED for v1 (too aggressive; ABSTAIN
   noise on syntactic rules). Default `false`, promote per rule.
4. CTP-native finding format instead of SARIF — REJECTED (fragments the
   ecosystem; SARIF is OASIS-standard and every corpus linter emits it).
5. Eager prose-judge on every MD every session — REJECTED (unbounded token
   cost). Cache + opt-in `CLAUDE_TDD_PRO_PROSE_JUDGE_EAGER=1`.
6. Reach across the contract boundary into the harness — REJECTED (CTP must not
   know its consumers; only `active.json` + `rubric/detectors/` move).

## 7. Verification (brief §7 acceptance, this CL)

- `prose-judge.sh` on the brief's exact example: an ADR section saying "leave
  dev-cluster ingress unrestricted (0.0.0.0/0)" → `violates`, SARIF result at
  the heading line, exit 1; a clean ADR → exit 0; cache populated; a
  keyword-mention-without-literal-and-no-model → `not_enforced` (exit 3).
- `md-structure.sh` via `enforce.sh --root <tree> --rule g-md-*`: a fence with
  no language → fail/red; a clean tree → pass/green; two H1s → fail; empty tree
  → not_applicable (no vacuous green).
- `md` namespace validates structurally (`validate-source-file`) identically to
  the existing IaC corpora; registered in `validate-all.sh KNOWN_NAMESPACES`.
- Active suite stays green (cl484-* added).
