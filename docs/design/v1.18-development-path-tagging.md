# v1.18 — Development-path tagging: every rule tagged for IaC, full-stack, or both (2026-06-30)

Operator directive: *"When a rule is tagged by the four-axis name registry, make sure it's tagged for
both IaC and the second development path (full-stack). Both development paths have their own rules, and
anything CTP generates must conform to both sets."*

## The need

CTP has two co-equal build flows (§28.62): IaC/cloud and full-stack application code. For "anything CTP
generates conforms to both sets" to be explicit and verifiable, every rule must be classified into the
development path(s) it governs — and a rule that governs both (a universal secret rule, a design rule,
shared config) must be tagged **both**.

## Why a naive split fails

The four-axis kind alone is not enough: IaC languages are *also* linguist aliases (HCL appears in
`linguist_aliases`, YAML/JSON are shared by both flows). A naive "has linguist → full-stack" would
mis-tag every terraform rule as application code. And 6 corpus rules carry no four-axis tag at all
(`g-universal-*`, `g-node-*`, `g-react-*`).

## The derivation (deterministic, from the four-axis + id)

`commands/classify-path.sh` derives `development_paths ⊆ {iac, fullstack}` for any rule:

- **iac** ← `iac_dialects` or `k8s_gvks` present, OR an IaC language (`hcl`) in `linguist_aliases`;
- **fullstack** ← an application language (`typescript`/`javascript`/`tsx`/`jsx`/`python`/`go`/`java`/
  `groovy`/…) in `linguist_aliases`;
- **both** ← `applies_to_prose` (design governs both flows); a cross-cutting rule
  (`universal`/`secret`/`license`/`dependency`/`supply-chain`/`provenance` in the id); or ambiguous
  config/markup (`yaml`/`json`/`markdown`) with no specific path;
- **namespace fallback** (untagged rule) — `g-node-`/`g-react-`/`g-ts-`/`g-js-` → fullstack;
  `g-aws-`/`g-gcp-`/`g-azure-`/`g-k8s-`/`g-hashicorp-`/`g-cfn-` → iac; otherwise → both.

This guarantees **every rule resolves to ≥1 path** (`--audit` exits 1 if any rule is unpathed). On the
current corpus: **total 118 → iac 42, fullstack 30, both 46, unpathed 0.** The 46 `both` rules are the
cross-cutting set that enforces on every artifact across both flows; the 42 iac + 30 fullstack are the
path-specific sets.

## How it composes

The path tag is the explicit partition behind §28.62 (co-design) and the planned both-paths pre-write
enforcement: a codesigned project's IaC artifacts are governed by the iac+both sets, its application
code by the fullstack+both sets — so anything CTP generates conforms to both rule sets, at every phase.
