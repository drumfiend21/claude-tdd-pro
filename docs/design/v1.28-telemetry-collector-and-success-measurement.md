# v1.28 — Telemetry collector, anonymous project identity, and software-success measurement (DESIGN)

**Status: DESIGN — operator-directed, backlogged for build.** Persisted per the operator's
instruction to store this as architectural design in the repo ahead of implementation
(mirrors the §31 "v1.22 DESIGN amendment" precedent — design lands first, build follows in
later CLs). Architecture pointer: §34. Built foundation this extends: §33 CL-564.n
(disclosed public-only source-URL telemetry) + `docs/telemetry.md`.

This document captures the operator's strategic vision and the principled design for it. It
is deliberately explicit about the consent model, because the value of the whole system
depends on it being trustworthy enough for mass adoption.

## 1. Vision — the self-improving flywheel

The plugins (`claude-tdd-pro` / the GCTP harness) can be given away free and open-source; the
durable value is the **data signal**: knowing which authoritative rule/standard **sources**
(URLs) correlate with the **most excellent and most successful** software, and feeding that
signal back to make GCTP/CTP produce better software over time.

```
 registered source URLs (Tier 0)
        +
 anonymous project identity (Tier 1)
        +
 engineering-health signals (Tier 1)
        +
 excellence-signature match vs a gold-standard corpus (Tier 1/2)
        +
 production/in-the-wild success (Tier 2, opt-in + barter)
        ↓
 correlate: which URL sets ↔ which excellence signatures ↔ which production outcomes
        ↓
 feed back into GCTP/CTP rule sourcing + generation → better software
        ↓
 (loop)
```

The flywheel only turns if contributors trust it. Therefore the design's first-class concern
is a **tiered consent model** keyed to data sensitivity.

## 2. Tiered consent model (the spine of the whole design)

Consent escalates with how private the data is. This is non-negotiable in the design: the
deeper the data, the stronger the consent, and anything touching private code or production
is **opt-in with a value exchange**, never default-on, never a rights grab.

| Tier | Data | Sensitivity | Consent | Status |
|---|---|---|---|---|
| **0** | Public source URLs (`scheme://host/path`, public-resolving hosts only) | Low (public facts) | Disclosed, **on by default**, opt-out (`CTP_TELEMETRY=off` / `DO_NOT_TRACK=1` / `telemetry: off`) | **BUILT** (§33 CL-564.n) |
| **1** | Anonymous project id + **aggregate** local engineering-health metrics (DORA + test/rule-pass/fitness), no code | Low–medium (numbers, no content) | Disclosed, on by default, opt-out (same switches) | DESIGN |
| **2a** | Production / in-the-wild success signals (uptime, adoption, deploy outcomes) | High (their running product) | **Opt-in + barter** (a perk/feature in exchange); wired at scaffold time | DESIGN |
| **2b** | Codebase-signature analysis (structure/quality patterns of their source) | Highest (their proprietary code) | **Opt-in + barter**, separate explicit toggle from 2a | DESIGN |

Principles binding every tier:

- **Disclosure always.** "Anonymous" reduces privacy risk; it never removes the duty to
  disclose. Every tier is documented in `docs/telemetry.md`.
- **Opt-in for anything private.** Tiers 2a/2b require affirmative opt-in — you cannot measure
  someone's private code or running product without their yes.
- **Barter, not coercion.** Opt-in tiers offer something back (added functionality, a perk,
  benchmark feedback: "here's how your project's health compares to the excellence corpus").
  The exchange is explicit and the base product stays fully functional without opting in.
- **No rights transfer.** Opting in grants permission to collect the specified telemetry —
  never ownership, license, or any claim over the user's code or product.
- **Anonymity by construction.** Random per-project UUID join key; no identity, account, repo
  name, or path required. Rotatable and deletable by the user.
- **License-clean + commercially distributable.** No copyrighted content (theirs or third
  parties') is ingested without a license check. The plugins remain sellable.

## 3. Anonymous project identity

- A **random UUID generated once per project**, stored at `.claude-tdd-pro/project-id`
  (gitignored by default; travels only if the user chooses).
- **Random, never derived** from repo path / git remote / org (those are reversible and would
  de-anonymize). It is an opaque **join key** only: it links a project's URL set to that
  project's health/success signals over time and reveals nothing about who or what it is.
- Rotatable (`project-id --rotate`) and deletable; deletion severs the join going forward.
- Reused across Tier 0/1 emits (already have `anon_install_id`; the project id is the
  per-repo analog, distinct from the per-install id).

## 4. Collector infrastructure (backend)

**GitHub is explicitly NOT the store.** It has no data-ingestion endpoint; using Issues/commits
as a database is fragile, rate-limited, and public. Use it for code only.

**AWS serverless — every resiliency property the operator asked for is a managed feature:**

- **Ingest:** API Gateway (HTTP API) → Lambda. This *is* the load-balancing + autoscaling
  (scales to zero idle, to thousands/sec under load; nothing to patch).
- **Store (recommended: S3 data lake for "collect now, mine later"):** Lambda → Kinesis
  Firehose → **S3**, partitioned by date. 11-nines durability; cross-region replication is a
  checkbox (redundancy); versioning = backup-of-backups; pennies at fleet scale. Analyze later
  with **Athena** (SQL over S3) — no running database to scale or pay for idle.
- **Store (alternative/companion: DynamoDB for live per-project lookups):** on-demand billing
  (auto-scale), automatic partitioning (**sharding** you never manage), Point-in-Time Recovery
  (**continuous backup**), Global Tables (**multi-region active-active replication**).
- **Best-of-both:** Firehose fan-out to S3 (durable lake) **and** DynamoDB (hot lookups).
- **Delivered as governed IaC.** The collector is generated as one-command Terraform/SAM
  **through this repo's own enforced IaC pipeline** (dogfooding), deployed with the operator's
  AWS credentials. Claude cannot provision live billable AWS from a session; it produces the
  deployable stack + the Lambda ingest handler, and the operator runs `terraform apply` /
  `sam deploy`, then points `CTP_TELEMETRY_ENDPOINT` at the API Gateway URL.
- **Client already fails open:** unreachable endpoint ⇒ silent no-op (§33 CL-564.n), so a
  restricted or offline session never breaks.

## 5. Measuring software success

Three complementary signals, increasing in richness and consent cost:

### 5.1 Gold-standard reference-corpus benchmarking (the "excellence signature")

Compare code built with GCTP/CTP against a corpus of **expert-verified excellent codebases**
plus their **expert reviews**, to extract a reusable *signature of engineering excellence*, then
score new plugin-built code by how strongly it matches that signature.

- **Candidate corpus: the O'Reilly Architectural Katas** — finalist submissions vetted by
  expert judges, with (believed) published judge reviews. This repo already references building
  a real "O'Reilly-kata submission" (§28.17 CL-477 context) and carries a kata north-star
  (SoftArchCert). **VERIFY BEFORE INGEST (open question O-1):** confirm online the exact set of
  finalist codebases, whether the judge reviews are public, and — critically — the **license**
  on both the codebases and the review text. Do not ingest copyrighted code or review prose
  without a compatible license; where licensing is unclear, derive only non-expressive metrics
  (structural/quality measurements), not copies.
- **Signature extraction:** from the corpus + reviews, distill measurable excellence markers —
  e.g. module cohesion/coupling, ADR quality + citation density (this repo already enforces
  prose ADRs, §28 CL-480), test pyramid shape, boundary discipline, security posture, naming
  and comment discipline. These become a scored **excellence-signature vector**.
- **Scoring:** a plugin-built project's code is scored for similarity to the excellence-signature
  vector. High similarity ⇒ "exhibits the qualities of excellence." This is computable on the
  plugin's own detectors + the aggregate metrics — **no need to exfiltrate source** (the
  signature score is a number; Tier 1). Deep per-file signature analysis is Tier 2b (opt-in).
- **Corpus is pluggable:** O'Reilly katas are one candidate source; any expert-verified set
  (award winners, canonical reference architectures) can seed the corpus, license permitting.

### 5.2 Local engineering-health metrics (already computed locally)

The plugin already computes health signals locally — reuse them as aggregate Tier-1 telemetry:

- **DORA:** deploy frequency, lead time, change-fail rate, MTTR (`security-posture.sh` already
  computes MTTR; the C-24 DORA scoreboard is the home).
- **Quality:** test pass rate, rule-pass rate, coverage trend, fitness-trend slope
  (`fitness-trend.sh`), cycle time (`cycle-time-bench.sh`).

Reported as numbers keyed to the anonymous project id + URL set — never source, file names, or
commit messages.

### 5.3 Production / in-the-wild success (Tier 2a, opt-in + barter)

Once a plugin-built app is deployed and live, opt-in production signals close the loop from
"looks excellent" to "succeeds in the real world."

- **Wired at scaffold time:** when GCTP/CTP generates the project, it can offer to include a
  small, clearly-labeled, opt-in telemetry beacon in the app.
- **The opt-in IS the feature (see §9).** The value is not a bolted-on perk traded for data —
  opting in *is what turns on* the benchmarking capability, because that capability is produced
  by the shared pool of opted-in projects. Contribution and benefit are the same act.
- **What "success" could mean (open question O-2, ideate further):** uptime/availability,
  deploy cadence sustained in prod, error/incident rates, adoption/traffic growth, rollback
  frequency, time-to-recover. All aggregate, all opt-in, all anonymous via the project id.
- **Freedom-preserving:** the app runs identically without the beacon; opting in grants no
  rights over the app or its data.

## 6. The feedback loop into GCTP/CTP

Correlate, across the fleet and over time:

- **URL sets ↔ excellence signatures ↔ production success.** Which authoritative sources are
  associated with code that both *scores* excellent and *succeeds* in production?
- Feed the winners back into: (a) the curated/community source registry (promote high-signal
  sources), (b) rule generation priorities, (c) the excellence-signature model itself.
- Result: GCTP/CTP become **self-improving** — each cohort of projects sharpens the next
  cohort's rules and generation. The correlation is the moat, even with the code given away free.

**Honesty caveat (O-3):** these are *correlations*. Teams that adopt rigorous sources may
already be rigorous (self-selection). Treat the signal as strong evidence to investigate, not
proof of causation; where possible, look for within-project before/after deltas (a project's
health trend after adopting a source) to approximate causal signal.

## 9. Consent-as-feature — the reciprocal data network effect

The strongest and most honest form of the Tier-2a exchange is not "give data, get an unrelated
perk." It is **structural reciprocity: the opt-in itself is the functionality.** The
benchmarking capability *exists because* participants share, so the act of contributing and the
act of benefiting are one and the same.

- **What the participant gets, the instant they opt in:** their standing within the fleet of
  software built with GCTP/CTP — percentile on each excellence / engineering-health / production
  dimension, their nearest anonymized exemplars, the specific gaps between them and the
  gold-standard excellence signature (§5.1), and concrete "do X to move up" guidance *derived
  from the higher-ranked cohort*. They learn how their product stacks up — immediately and
  continuously — which is exactly the reward, and it is the same data event that feeds the
  author's corpus.
- **Data network effect (the compounding moat):** more opt-ins → tighter percentiles, richer
  exemplars, sharper "do X" guidance → a stronger reason for the next project to opt in. Each
  participant makes the benchmark better for every other participant. This is a two-sided value
  loop, not a one-way extraction — and it is defensible even with the code given away free,
  because the *aggregate* is the asset and no competitor holds it. The author may keep the
  dataset proprietary or publish it; either way the network effect is the moat.
- **Author's side:** a unique, growing corpus of what excellent, successful software looks like
  and which sources/patterns produce it — the "gold mine" that survives open-sourcing the code.
- **Honesty + guardrails (non-negotiable):** it remains a genuine, disclosed exchange; the base
  product is fully functional without opting in (never a dark pattern, never degrade the
  non-participant); anonymity is preserved; **no comparative statistic is shown against a cohort
  below a k-anonymity floor** (never reveal a "cohort of one"); no rights transfer. Reciprocity
  is the reason it can be honest *and* valuable at once: the user is not paying with their data
  for something else — their participation is the product they receive.

## 10. Meta-improvement engine — from correlation to automated product + process improvement

The point of the data is not to admire it but to **reliably deduce what improves software, act
on it automatically, and improve the act of improving itself.** Three layers, each with honest
guardrails.

### 10.1 Inference ladder — strengthen deduction beyond naive correlation

Climb from "associated with" toward "causes," raising confidence at each rung:

1. **Correlation** — baseline (which URL sets co-occur with high excellence/production scores).
2. **Within-project before/after deltas** — a project's health/production trend *after* it
   adopts a source or rule (the project is its own control; removes much self-selection bias).
3. **Cohort / quasi-experiment** — matched projects (same stack/domain/size) that did vs did not
   adopt a source; compare outcome distributions.
4. **Interrupted time series** — model the trend, detect the step-change at the adoption event.
5. **Deliberate experiments** — canary cohorts / holdouts / A-B on a rule or generation-heuristic
   change; uplift modeling to estimate who benefits. This is the top rung: real causal signal.

Every claim carries its rung, effect size, confidence, and cohort n — so downstream action is
weighed by evidence strength, never by a raw correlation alone.

### 10.2 Action layer — automate product improvement (human-gated)

Validated patterns become *proposed* actions, drafted automatically with their evidence
attached: promote high-signal sources into the curated registry, deprecate low/negative-signal
ones, propose new rules or generation heuristics, retune the excellence-signature weights. **All
PR-gated (§31.2) / human-in-the-loop** via the existing review-queue and GCTP self-healing
(TICKET-008) — the engine drafts a decision with evidence; a human approves. **Never a silent
mutation of the standards.** This is what "ease + automate the improvement process" means: a
reviewer reviews a well-evidenced decision, not a pile of raw data.

### 10.3 Meta-learning — improve the improvement

Each automated change is itself tagged and its *downstream* effect measured: did promoting
source X actually raise the adopting cohort's health, or not? The engine thereby learns which
*kinds* of interventions pay off, tracks an **improvement hit-rate** over time, and reallocates
effort toward high-yield intervention types. The loop closes on itself — GCTP/CTP get better at
getting better.

### 10.4 Guardrails against feedback pathologies (stated, not glossed)

- **Goodhart's law** — once a metric is optimized it stops measuring what it did. Mitigate by
  triangulating multiple, rotating metrics and watching for gaming/overfit to the signature.
- **Self-selection / survivorship bias** — prefer within-project deltas + matched cohorts over
  raw cross-sectional correlation; account for projects that vanish (survivorship).
- **Feedback echo chamber** — the engine can reinforce its own past picks. Reserve an
  **exploration budget** (periodically re-test deprecated/low-ranked sources) so the corpus does
  not ossify around early winners.
- **Re-identification in aggregates** — k-anonymity floors plus differential-privacy noise on any
  published/comparative statistic; never a stat over a sub-threshold cohort.
- **Over-fitting one ecosystem** — segment analysis by stack/domain; a pattern true for Node web
  apps may not hold for embedded C — do not globalize a segment-local finding.
- **Reproducibility** — version the model, the signature weights, and dated data snapshots so any
  automated decision can be re-derived and audited.

**Open questions this adds:** **O-6** — the inference-ladder methodology + minimum evidence rung
required before an automated action is *drafted* (vs merely logged); **O-7** — the
exploration-vs-exploitation budget split for the meta-learner; **O-8** — the k-anonymity floor +
differential-privacy parameters for any comparative/published statistic.

## 11. Repo ownership — where each item is built (operator-directed determination)

Per the prime directive (CTP is the **capability / standards provider**; GCTP is the
**orchestration / app-scaffolding / user-facing consumer**), and the joint-design regime
(cross-plugin items may be designed together but are still committed to their own repos), each
roadmap item is assigned a build location. **Author-infra** = the operator's telemetry backend:
neither plugin's runtime, generated *via* the governed IaC pipeline but deployed and operated by
the author (so it never becomes a consumed plugin surface and does not violate the prime
directive). **Joint** = spans both repos; the CTP half and GCTP half land as separate CLs in
their own repos.

| # | Item | Build location | Rationale |
|---|---|---|---|
| 0 | Tier-0 public-URL telemetry | **CTP** — BUILT (CL-564.n) | Part of the CTP standards-refresh source lifecycle; emits regardless of harness. |
| 1 | Tier-1 client: anon project-id + aggregate health/DORA emit | **CTP** | DORA/health signals are CTP capabilities (`security-posture.sh`, `fitness-trend.sh`); extends the CL-564.n telemetry framework. |
| 2 | Collector IaC (API GW → Lambda → S3 lake / DynamoDB) | **Author-infra** (generated via CTP's governed IaC pipeline) | The operator's backend, not plugin runtime. Recommend a dedicated deployment repo or a CTP `infra/collector/` deliverable that is explicitly NOT part of the consumed plugin surface. Operator confirms deployment home. |

| 4 | Tier-2a production beacon + barter | **Joint / split** | Beacon **data contract** = CTP capability. **Scaffold-time injection** into the user's generated app = GCTP (project generation is GCTP's outer loop). |

joint items (3, 4, 6, 7, 8) are exactly the boundary-contract work the joint-design regime
exists for — each will land its CTP half in CTP and its GCTP half in GCTP, coordinated but never
cross-staged. The GCTP-owned slice is also recorded in the GCTP repo
(`docs/backlog-telemetry-success-measurement.md`) pointing back to this document as the
authoritative design.
