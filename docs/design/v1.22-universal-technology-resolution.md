# v1.22 — Universal technology resolution + dynamic rule-sourcing (§31)

**Operator directive (2026-07-05):** "There needs to be a mechanism to pick up all technologies and
categorize them under an appropriate umbrella like 'frontend' to look up the applicable rules… the plugin
searches from the same rule sources it has for rules concerning the particular technology requested by the
user or determined to be most effective. For example, React may not always be the best solution… Angular
might be."

**Status:** DESIGN (not yet built). Additive, append-only. Feature IDs **S-58…S-62**, contracts **§2.36 /
§2.37**, section **§31**.

## 0. TL;DR

Today the classifier maps a vision to a **finite signal catalog** → workload umbrellas → the 44 namespaces
that already exist. It misses any technology it has no signal for (Vue, Angular, Django, Go, Rust…), and it
can only apply rulesets that already exist. This design makes technology capture **open-ended** by resolving
any named technology to its **canonical 4-axis coordinate** (Linguist / PURL / K8s-GVK / IaC), classifying
that coordinate under an **umbrella**, applying the umbrella's general rules immediately, and — when a
tech-specific ruleset is missing — **acquiring it through the existing scrape→tag pipeline** (the same URL
sources + fetchers + 4-axis tagging), with cite-or-decline when no authoritative source exists. It then
**recommends the best-fit technology** per umbrella (Angular vs React) and applies the chosen tech's rules
across all three lifecycle stages: consult, architectural design, code generation.

## 1. The existing pipeline (grounding — do not re-invent)

```
scrape                         tag (4-axis)                 aggregate            look up + apply
standards/*sources*.yaml  →   namespace-axis-binding.yaml → rubric/aggregator → classifier (vision→umbrella→ns)
+ standards/fetchers/*         (linguist / purl / gvk / iac)  118 rules /          → §30 intake probes + grounding
(html-anchor, markdown-                                       44 namespaces        → §29 enforce (native + FOSS tools)
 headers, pdf-section)                                                             → §29.6 write-time (byte-identical)
```

The **4-axis kind registry** (`standards/kind-to-tool-routing.yaml`: `linguist_aliases`, `purl_uses`,
`iac_dialects`, k8s GVK) is the canonical tagging layer, and `namespace-axis-binding.yaml` binds each
namespace to its axis coordinate + the FOSS tools that enforce it. **This design extends that pipeline; it
does not replace any of it.**

## 2. Design invariant — the 4-axis registry IS the universal capture mechanism

Every technology has a canonical identity in one of the four registries:

| Named by user | Canonical 4-axis coordinate | Umbrella |
|---|---|---|
| Vue | `linguist:Vue` · `purl:pkg:npm/vue` | frontend |
| Angular | `linguist:TypeScript` · `purl:pkg:npm/@angular/core` | frontend |
| Django | `linguist:Python` · `purl:pkg:pypi/django` | backend-web |
| Rust/Tokio | `linguist:Rust` · `purl:pkg:cargo/tokio` | backend-language |
| Helm chart | `iac:helm` | orchestration |
| CRD | `gvk:<group>/<version>/<kind>` | orchestration |

So capture is **not a hand-maintained list of blessed technologies** — it is a *resolution* against
open-ended registries. Any package in a known ecosystem (npm/pypi/cargo/gem/maven/go/nuget…) resolves; the
umbrella is derived from the axis (PURL ecosystem + Linguist type → language/framework umbrella; IaC dialect →
iac umbrella; GVK → orchestration). The umbrella determines the **general** rule namespaces that always apply;
the coordinate determines the **specific** namespace to apply or acquire.

## 3. Feature decomposition (S-58…S-62)

- **S-58 Technology resolver** — `commands/resolve-technology.sh <name>`: resolves a named technology to its
  canonical 4-axis coordinate + umbrella, and reports `namespace=<ns>` (specific ruleset exists) or
  `needs_source=<coordinate>` (recognized, umbrella-classified, specific ruleset absent). Cite-or-decline: a
  name that resolves to no registry coordinate → `unresolved` (never guessed). Contract **§2.36**.
- **S-59 Umbrella taxonomy registry** — `standards/technology-umbrella-registry.yaml`: maps 4-axis coordinate
  patterns → umbrella → `{ general_namespaces: [...], specific_namespace: <ns|null> }`. `general_namespaces`
  are the umbrella-wide rulesets (frontend → `owasp, web-vitals`, + `wcag` at accessibility posture; backend →
  `owasp, twelve-factor-app`; iac → `security-governance`). Additive; extends the classifier, does not replace
  it.
- **S-60 Dynamic rule-source acquisition** — `commands/acquire-technology-rules.sh <coordinate>`: for a
  `needs_source` technology, resolves an **authoritative** source (a curated `technology-source-registry.yaml`
  maps coordinate → candidate authoritative URL, e.g. `purl:pkg:npm/vue → vuejs.org/style-guide`), fetches it
  through the **existing fetchers**, parses into rules, **tags each with the 4-axis coordinate**
  (`namespace-axis-binding`), attaches provenance (source id + url + `fetched_at` + tier), mints the namespace,
  and re-aggregates. Contract **§2.37**. Cite-or-decline: no approved authoritative source → stays
  `needs_source`; umbrella-general rules still apply; **rules are NEVER invented**.
- **S-61 Technology-fitness recommender** — extends S-34 `architect-recommend`: for an umbrella with multiple
  candidate technologies, scores each against the workload profile (rendering need, scale, consistency,
  team-familiarity, ecosystem maturity), grounded in comparative sources, and recommends the best-fit — so the
  consult can propose Angular over React when the workload favors it. The operator accepts or overrides; the
  chosen technology's ruleset is then resolved/acquired (S-58/S-60) and applied. Cite-or-decline: no
  ungrounded preference.
- **S-62 Full-lifecycle rule application** — composition (no new command): a resolved/acquired technology's
  rules apply at **consult** (S-57/§30 probes + grounding), **architectural design** (§29.4/§29.6 enforcement
  of the produced design), and **code generation** (§29.6 byte-identical write-time — the 4-axis linguist/iac
  tag routes the rule to the right generated files and the `foss_tools` binding routes to the right enforcer).

## 4. Contracts

- **§2.36 Technology-resolution contract.** Every named technology resolves to exactly one of:
  `{ namespace }` (specific ruleset present), `{ needs_source, coordinate, umbrella }` (recognized, general
  rules apply, specific ruleset acquirable), or `{ unresolved }` (no registry coordinate — declined, not
  guessed). The umbrella's `general_namespaces` are ALWAYS applicable to a resolved technology, regardless of
  whether the specific ruleset exists.
- **§2.37 Dynamic-source-acquisition contract.** A new ruleset enters the surface **only** through the §2.6
  source path: an authoritative URL in a source catalog, fetched by a registered fetcher, tagged with a 4-axis
  coordinate, carrying provenance (`source`, `url`, `fetched_at`, `tier`). No rule is ever hand-authored or
  inferred for a technology; the freshness/tier gates (§2.6) apply unchanged. Acquisition is gated on an
  **approved authoritative source** — the honest limit that keeps "rules come from real sources" true.

## 5. Application across the three lifecycle stages

1. **Consultation** — S-58 resolves each technology in the vision/answers/stack; in-scope namespaces (specific
   + umbrella-general) drive S-57 probes and §29 grounding. `needs_source` technologies are surfaced (mirror
   of §30.7 reveal), optionally triggering S-60 acquisition.
2. **Architectural design** — the produced design is reasoned + natively enforced against every applicable
   namespace (§29.4/§29.6), including any newly-acquired one, since the aggregator picked it up.
3. **Code generation** — generated files are enforced byte-identically at write-time (§29.6); the 4-axis tag
   routes each rule to the files of its kind and to its `foss_tools`, so an acquired `vue` ruleset lints `.vue`
   files exactly as `react` lints `.tsx`.

## 6. Honest boundaries (explicit, non-negotiable)

- The system can **always** resolve a technology to a coordinate, classify its umbrella, and apply the
  **umbrella-general** rules.
- It can apply **tech-specific** rules only when an authoritative source exists or is acquired. Dynamic
  acquisition requires an **approved authoritative source** (S-60's source registry + §2.6 tier gate) — it
  cannot conjure rules for a technology with no published authoritative guidance. Such technologies remain
  `needs_source`: recognized and umbrella-covered, specific-ruleset pending.
- Technology-fitness recommendations are **grounded or declined** — no unbacked "use X over Y."

## 7. Compatibility + rollout

Fully additive. The existing classifier signal catalog remains as a fast path; S-58 resolution is a superset
that also covers un-signalled technologies. v1.0/v1.1 profiles and every existing namespace are untouched. The
surface only grows (new namespaces via S-60); no existing rule/contract changes. Suggested sequencing (§20):
S-58 resolver → S-59 taxonomy → S-60 acquisition → S-61 recommender → S-62 lifecycle composition.

## 8. Acceptance sketch (per feature, ≥10 specs each at build time)

- S-58: resolves React/Vue/Django/Rust to correct coordinate+umbrella; `needs_source` for umbrella-known but
  ruleset-absent; `unresolved` for a non-technology; umbrella-general namespaces always returned.
- S-59: coordinate→umbrella→general_namespaces mapping; frontend general rules apply to Vue with no vue ns.
- S-60: acquires a ruleset from an authoritative source through the real fetcher; tags 4-axis; provenance
  present; cite-or-decline with no approved source; aggregator picks up the minted namespace.
- S-61: recommends best-fit tech per umbrella grounded in comparative sources; override honored; no ungrounded
  preference.
- S-62: an acquired ruleset enforces at write-time on its file kind byte-identically (composes §29.6).
