# v1.27 — standards-refresh domain-crawl (§33 CL-564.f)

**Status: BUILT (2026-07-12).** Extends the §33 registry-walker orchestrator (design:
[v1.26-standards-refresh-registry-walker.md](v1.26-standards-refresh-registry-walker.md)) so a
single registry entry can govern a bounded, guarded, audited crawl of its documentation
domain instead of one page. Operator-approved 2026-07-12 (item 5 of the GCTP handoff §4
extension roadmap, built with the proposed defaults).

## Why

Authoritative guidance rarely lives on one page. `nodejs.org/api/` is ~50 module pages;
WCAG techniques, OWASP cheat-sheets, cloud provider well-architected docs are all page
families. Before this CL every registry entry mapped to exactly one fetched URL, so
whole-source acquisition (§31.9) under-collected by construction for multi-page sources.
The alternative — enumerating every page as its own registry entry — duplicates registry
maintenance and still goes stale when upstream adds a page.

## Shape

A registry entry MAY declare a `scope:` block:

```yaml
- id: nodejs-api-docs
  url: https://nodejs.org/api/
  fetcher: html-anchor.sh
  scope:
    strategy: path-prefix        # single-page | path-prefix | sitemap | link-follow
    max_pages: 100               # tightens the hard cap (200); never exceeds it
    max_depth: 2                 # tightens the hard cap (3); never exceeds it
    respect_robots: true         # default true (http/https only)
    rate_limit_ms: 1000          # default 1000 between network fetches; file:// exempt
    include_patterns: []         # substring allowlist (empty = all)
    exclude_patterns: []         # substring denylist, EXTENDS the built-in defaults
```

No `scope:` ⇒ `single-page` ⇒ the pre-existing one-URL path, byte-identical.

`commands/domain-crawl.sh` owns the expansion: seed page (already fetched through the S-2
`fetcher.sh` + `http-get.sh` discipline, cache/fragility semantics untouched) → BFS/leaf
expansion per strategy → Stage 1 extraction per fetched page → merged,
content_hash-deduped segments JSON on stdout. The orchestrator consumes that exactly where
a single `extract-rules-from-url.sh` call's output went before, so classify → route →
prose-forcing → Stage 5 fidelity → sufficiency → assemble → merge/deprecate → atomic
write are all unchanged and apply to crawled sources automatically.

## Strategies

- **path-prefix** — follow `<a href>` links whose resolved URL shares the seed's host and
  whose path starts with the seed's directory prefix (`/api/` for `…/api/index.html` or
  `…/api/`). The canonical docs-tree shape (Node, MDN sections, provider docs).
- **sitemap** — the seed IS a sitemap index: its `<loc>` URLs are fetched as depth-1
  leaves; the index itself is never extracted; leaf links are not followed.
- **link-follow** — same-host BFS with no path constraint, for docs sites whose sections
  interlink across prefixes. The only strategy allowed to span a whole host, and only by
  explicit declaration.

## The 8 universal guards (always on; entry values only tighten)

1. **same-host** — a link off the seed's scheme+host is never enqueued (`off-host`).
2. **max_pages ≤ 200** hard cap; overflow URLs recorded as `max-pages`.
3. **max_depth ≤ 3** hard cap.
4. **robots.txt** — `User-agent: *` `Disallow:` prefixes honored for http/https seeds;
   unfetchable robots ⇒ allow (fail-open on the guard, bounded by the other seven).
5. **content-type filter** — binary-extension excludes + NUL-byte content sniff
   (`binary-content`).
6. **content_hash dedupe** — identical page bodies processed once (`duplicate-content`);
   segment-level dedupe additionally collapses identical rules across pages.
7. **rate limit** — `rate_limit_ms` sleep (default 1000) before each network fetch;
   `file://` fixtures exempt so the suite stays fast.
8. **URL length ≤ 2048** bytes (`url-too-long`).

**Refuse-on-root-path:** `path-prefix` with a root (`/`) prefix exits 2
(`REFUSED … whole-domain crawl must be an explicit link-follow choice`). The orchestrator
surfaces it as an entry failure.

## Quality guards

- **empty-content-density** — pages with fewer than 5 words of tag-stripped text are
  skipped (`empty-content`), so nav shells and redirect stubs don't become rules.
- **low-yield** — a fetched page yielding 0 segments is recorded (`low-yield`) but does
  not fail the crawl.
- **default exclude_patterns** — login/signup/signin/register/search/mailto/javascript
  URLs and static assets (pdf/zip/tar/gz/images/css/js/fonts/video) are excluded before
  any entry-declared patterns are added.
- **confidence-density audit** — the per-entry pipeline summary now carries
  `confidence_density=<0.00–1.00>` (fraction of emitted rules with high classifier
  confidence), so a low-quality crawl is visible at the summary line.
- **redirects** — following is delegated to the S-2 `http-get.sh` stub (`curl -L`).
  Link EXPANSION can never leave the host (guard 1). Known bound, recorded honestly: a
  same-host page 30x-redirecting off-host contributes its landing content (not its links);
  the binary/density/robots guards still apply to it.
- **crawl manifest** — every URL considered gets one JSONL row
  (`url`, `depth`, `status: fetched|skipped|index`, `reason`, `content_hash`, `segments`)
  at `.claude-tdd-pro/crawls/<source-id>-<ts>.jsonl`, one file per refresh run
  (deterministic under `--now`). The crawl is auditable after the fact without re-running.

## What this does NOT change

- Seed fetch discipline (S-2 fragility tiers, cache preservation) — untouched.
- Freshness windows — the entry is still one freshness unit keyed on its `id`.
- §28.40 grandfathering, deprecation-on-removal, §31.9 sufficiency, Stage 5 fidelity,
  Stage 6 gating — all downstream and unchanged; they now simply see more segments.
- Registry schema for existing entries — `scope:` is additive; every registry on disk
  parses exactly as before (§28.40 schema-additive discipline).
