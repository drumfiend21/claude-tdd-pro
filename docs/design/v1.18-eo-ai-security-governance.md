## §28.0 Source of the amendment — the executive order

On **2026-06-02** the White House issued the executive order **"Promoting Advanced Artificial
Intelligence Innovation and Security"** (published in the Federal Register 2026-06-05). Its
stated policy is U.S. AI leadership through an **innovation-first, voluntary** posture —
**Sec. 1 expressly disclaims any mandatory licensing, preclearance, or permitting requirement**
for developing, releasing, or distributing AI models — paired with stronger cyber defenses. The
operative sections, as corroborated across the primary text and multiple independent legal
analyses (Covington/Inside Privacy, A&O Shearman, Hogan Lovells, DLA Piper, Wiley, Skadden,
Sidley, Crowell, WilmerHale, Tenable):

| EO section | Title | Core directive |
|---|---|---|
| **Sec. 2** | Upgrading American Systems for Advanced AI | CNSS prioritizes cyber defense of National Security Systems (30 days); **Treasury + NSA + CISA stand up an "AI Cybersecurity Clearinghouse" (30 days) focused on finding and fixing software vulnerabilities**. |
| **Sec. 3** | Secure Frontier Model Deployment | Classified **benchmarking** of advanced cyber capabilities (60 days); a threshold defining a **"covered frontier model"**; a **voluntary framework** giving the government **up to 30 days of pre-release access**, subject to **confidentiality, cybersecurity, insider-risk, and IP protections**, plus trusted-partner early access. |
| **Sec. 4** | Protection Against Criminal Actors | Attorney General prioritizes criminal enforcement against **AI-enabled cyber activity** — illegal computer access, identity fraud, wire fraud. |

Deadlines anchor the urgency: 30-day deliverables by **2026-07-02**, 60-day by **2026-08-01**.

**Why this is in scope for Claude TDD Pro.** The plugin is a test-first software-development
harness whose existing thesis is *grounded, cited, auditable, enforced-in-three-surfaces*
engineering. The EO's cyber-defense, vulnerability-remediation, provenance/IP-protection, and
voluntary-governance themes map directly onto surfaces the plugin already owns (the §2.8 AI
Provenance Manifest, the §2.9 control mapping, the O-6 signed audit checkpoints, the §2.24
portable audit-pack, the S-2 fetchers, and the §27.28/§27.29 three-surface enforcement pattern).
This amendment **composes** those primitives into EO-aligned features; it invents no new engine.

**Explicit scope boundary (anti-overreach).** The plugin does **not** run frontier models,
operate the federal clearinghouse, or make "covered frontier model" determinations — those are
government functions. Every feature below is a **developer-side governance/engineering surface**:
it helps an org *engage* the voluntary framework and *harden* its own software, grounded in
public authorities (NIST/CISA/SLSA/OpenSSF) with the plugin's standing **cite-or-decline**
discipline. Sec. 4 (DOJ enforcement) is not a software feature; its repo-relevant residue
(misuse-resistant audit trails) is folded into C-22's control table, not given a standalone ID.

## §28.1 Feature decomposition (EO→mechanism)

All IDs are the next free number in their phase (verified against the architecture: H≤13, S≤53,
C≤21, X≤9; contracts ≤§2.29). Each feature reuses existing substrate where the "reuses" column
names it.

### Phase H — cyber-defense hardening (EO Sec. 2 + Sec. 3 IP protections)

- **H-14 — Dependency vulnerability scan + remediation gate.** `commands/vuln-scan.sh` reads the
  project's declared dependencies (`package.json`/`package-lock.json`, `requirements.txt`,
  `go.mod`, `Cargo.toml`), matches them against an **injected, deterministic advisory dataset**
  (fixtures, exactly as S-21 injects HTTP responses and the platform boundaries inject native API
  responses — determinism is preserved; no live network in the test substrate), classifies each
  finding by CVSS severity, and **gates** per §2.30: `critical`/`high` block (exit 1), `medium`/
  `low` warn, none ⇒ green no-op (exit 0). Emits a remediation report naming the fixed-in version
  for each finding (the EO Sec. 2 "find **and fix**" loop). Grounded in CISA SSDF / NIST 800-53
  (S-54). *Reuses:* §2.2 detector exit-code contract, §2.4 `security` eval category.
- **H-15 — SBOM generation + signed provenance attestation.** `commands/sbom.sh` generates a
  **CycloneDX** (default) or **SPDX** software bill of materials for the build artifact and
  produces a **SLSA-style signed provenance attestation** linking the SBOM digest to the §2.8 AI
  Provenance Manifest. Supports the EO's "protect American ingenuity and IP" theme and Sec. 3's
  IP-protection condition on pre-release access. *Reuses:* O-6 audit-log signing key handling
  (H-4 threat model), the `slsa` namespace already in the §1 inventory, §2.8 manifest.
- **H-16 — Frontier-model pre-release governance readiness checklist.** `commands/frontier-eval.sh`
  emits an **operator-readable readiness scaffold** for an org choosing to engage the EO Sec. 3
  voluntary framework: a cyber-capability **self-assessment checklist** and the four EO-named
  pre-release control families — **confidentiality, cybersecurity, insider-risk, IP-protection** —
  each slot grounded (cite-or-decline) in NIST AI RMF (S-54). It is a **governance checklist, not
  a model evaluator**; a control with no grounding source is marked `needs_grounding` (mirrors the
  S-8/S-26 decline contract). *Reuses:* the S-26 pillar-review scaffold shape, cite-or-decline.

### Phase S — grounding sources (the fuel for H-14/H-15/H-16/C-22)

- **S-54 — EO security-governance source catalog.** Secures + registers the EO-relevant tier-1
  authorities **not already in the catalog** into a new `standards/eo-security-sources.yaml`
  (NIST SP 800-53/171/RMF are already secured per §27.14 and are reused, not re-added). New seed
  sources (URLs verified at authoring time, `applies_to: security-governance`): `cisa-ssdf`
  (NIST SP 800-218 Secure Software Development Framework), `nist-ai-rmf` (NIST AI 100-1 AI Risk
  Management Framework), `slsa-framework` (SLSA supply-chain levels), `openssf-scorecard`
  (OpenSSF Scorecard checks), `cisa-kev` (CISA Known Exploited Vulnerabilities catalog). *Reuses:*
  the §2.6 two-tier source schema, the S-23/S-30/S-31 "own catalog file, registered namespace"
  pattern (keeps the S-1 "exactly 17 default sources" baseline untouched).

### Phase C — compliance & governance mapping (EO Sec. 2/3/4)

- **C-22 — EO control-mapping + national-security-systems compliance profile.** Registers an EO
  control mapping in the §2.9 schema (`framework: eo-advanced-ai-2026`, control ids `eo-sec2-*`,
  `eo-sec3-*`, `eo-sec4-*`) and adds a `national-security-systems` profile extension layered over
  the existing `government` profile (§2.5). The mapping ties each EO directive to the plugin
  control that satisfies it (Sec. 2 ⇒ H-14/H-15; Sec. 3 ⇒ H-16/H-15; Sec. 4 ⇒ misuse-resistant
  audit trail = C-4 + L-14). Cross-walked to NIST AI RMF, SSDF, and NIST 800-53. *Reuses:* §2.9
  control mapping, §2.5 profile system.
- **C-23 — Coordinated vulnerability-disclosure evidence record.** `commands/cvd-record.sh`
  produces a **clearinghouse-style coordinated-disclosure record** from H-14 findings — the EO
  Sec. 2 "find and fix" loop made auditable: `{advisory_id, severity, component, fixed_in,
  disclosed_at, remediation_status, source}` — and feeds it into the C-4 audit log and the §2.24
  portable audit-pack so it is **exportable for government/agency review**. *Reuses:* C-4 audit
  log, §2.24 audit-pack, L-14 audit-integration pattern.

### Phase X — execution surfaces (three-surface enforcement)

- **X-10 — Vulnerability + SBOM CI gate.** Wires H-14 and H-15 into the closed-loop CI workflow
  (`.github/workflows/closed-loop.yml`) and the `/doctor` surface with the **same exit-code
  contract**, exactly as §27.28/§27.29 wired S-30 into `/doctor` + CI. Safe-by-default: a repo
  with no declared dependencies and no IaC is a green no-op (cannot turn a previously-green repo
  red). *Reuses:* the §27.28/§27.29 `/doctor --check ... --root .` + CI-step pattern, X-1 GitHub
  Actions surface.

## §28.2 Cross-cutting contracts

- **§2.30 — Vulnerability-remediation gate contract.** Defines the find→classify→remediate→record
  loop. Severity thresholds: `critical`/`high` ⇒ block (exit 1), `medium`/`low` ⇒ warn (exit 0
  with marker), none ⇒ green no-op (exit 0). **Safe-by-default:** a project with no resolvable
  manifest is `vuln-scan no_dependencies=true status=skipped` (exit 0). Every blocked finding
  names its `fixed_in` remediation (the "and fix" half is mandatory, not advisory). Findings are
  recorded via C-23. Observable markers: `vuln-scan status=green deps=<n> vulnerabilities=0`;
  per-finding `vuln-scan component=<c> severity=<s> fixed_in=<v>`; summary `vuln-scan status=red
  critical=<n> high=<n>`. The advisory dataset is injected (deterministic substrate); the live
  feed is a production edge (mirrors the §27.19 external-call-safety discipline).
- **§2.31 — SBOM + signed-provenance attestation contract.** Every SBOM is emitted in a named
  standard format (`cyclonedx` default | `spdx`) with a `schema_version`, the component inventory,
  and a content digest. The signed provenance attestation binds `{sbom_digest, artifact_digest,
  builder, materials, signature}` and links to the §2.8 AI Provenance Manifest. Signing reuses the
  O-6 / H-4 key-handling trust model; an unsigned run is marked `attestation=unsigned` and never
  claims provenance. SLSA-level claims are grounded in the `slsa-framework` source (cite-or-decline).
- **§2.32 — Frontier pre-release governance contract.** The H-16 readiness checklist MUST cover the
  four EO-named control families — `confidentiality`, `cybersecurity`, `insider_risk`,
  `ip_protection` — for the up-to-30-day pre-release access window. Each control is grounded
  (cite-or-decline) in a secured authority (NIST AI RMF / NIST 800-53); an ungrounded control is
  `needs_grounding`. The checklist is **advisory and voluntary** — it emits no determination, sets
  no threshold, and explicitly records the EO Sec. 1 no-mandatory-licensing disclaimer in its
  header so the artifact cannot be mistaken for a compliance mandate.

## §28.13 Universal citation-conformance (contract §2.33, operator-directed, 2026-06-15)

**Operator-directed: widen and deepen the tests so that ALL architecture, design, and code the
plugin produces — full-stack AND cloud — conforms to cited rules from all the plugin's authoritative
sources.** Scope chosen by the operator: the broadest reading — every artifact cites a grounded
source spanning BOTH source triads. This is one standing invariant, enforced by a deterministic
conformance auditor + a deepened active-suite spec battery. No new feature ID; new contract §2.33.

### The two source triads (all six are in scope)

| Triad | Members | Grounds |
|---|---|---|
| Operator registries (the §1 through-line) | **STANDARDS**, **COMPLIANCE**, **PR-CORPUS** | full-stack coding rules, regulatory controls, peer-review patterns |
| Cloud grounding catalogs | **cloud-architecture (S-23)**, **engineering (S-30/S-31)**, **EO-security (S-54)** | cloud architecting / design / build |

### Producing surfaces audited (cite-or-decline on each)

| Surface | Where | Citation shape |
|---|---|---|
| Coding rules (full-stack) | `generated-code-quality-standards/<ns>/*.yaml` | `provenance[]` carrying `source` + `class`; reading-sources carry a populated `source:` header |
| Cloud convention rules | `standards/cloud-conventions/*.yaml` | per-rule `source_id` ∈ (S-23 ∪ S-30/S-31 ∪ S-54) |
| Translated design concerns | `business-translate.sh` output | `source_id` per concern, else `needs_grounding` |
| Architect recommendations / ADRs | `architect-recommend.sh`, `cloud-adr.sh` | `source_id` / pillar grounding, else `needs_grounding` |
| Vulnerability findings | `vuln-scan.sh` (H-14) | `source` per finding (an advisory authority) |

### Enforcement

`rubric/detectors/audit-citation-conformance.sh` — deterministic, `--root <dir>`; for each surface it
verifies every produced item cites a grounded source from the appropriate source system; emits a
per-surface marker `citation-conformance surface=<s> status=<green|red> items=<n> ungrounded=<m>` and
an overall verdict; exit 0 green / 1 red / 2 usage. The active-suite battery (CL-472) WIDENS coverage
(full-stack coding + cloud convention + all three registries present + all three cloud catalogs active
in the grounded set) and DEEPENS it (per-catalog grounding proof; negative paths — an injected
ungrounded rule/finding is caught; the auditor is green on the conformant tree and red on an injected
violation). This is the runnable generalization of §25 (pending-spec fidelity), §27.27 (cloud-output
conformance), and the S-30/S-8 cite-or-decline discipline.
