# v1.18 — §16 config plane threaded into §28 enforcement (design detail for architecture §28.44)

> Detail file for the append-only architecture reference block `§28.44`. The constitution carries
> only the extractable summary; rationale + the ground-assessment gap table live here per the
> amend-by-reference discipline.

## The question this answers

Are the ESLint-style **configuration plane** (§16: E-1 per-config severity / enable-disable, E-3
glob overrides, E-5 inline suppression) and the **4-axis labeling + enforcement plane** (§28) in
conflict, and can the config plane run *before* enforcement as one ordered pipeline without
regressing the suite? **Answer: no conflict — they are orthogonal layers on the same rule object,
and the ordering is `resolve-config → (4-axis) route → enforce → grade`.** This CL closes the first
gap: the config plane now governs the native per-rule enforcement path.

## Ground assessment (read-only audit, before any substrate)

Every rule already carries **both** planes on one object — config-plane fields (`severity`,
`rule_state`, `recommended`, `options_schema`) **and** routing-plane fields (`applies_to`,
`enforced_by`). The resolvers existed but were **not wired into enforcement**:

| §16 affordance | resolver | native path (`enforce-file.sh`) before | external path (`composite-dispatch`) |
|---|---|---|---|
| Severity (E-1) | `resolve.sh`/`active.sh` | used **static** rule severity, not resolved | 0 severity concept |
| Enable/disable (`off`) | `active.sh --emit-resolved` | ran **all** glob-matching rules | none |
| Glob overrides (E-3) | `active.sh --for-file` | not applied | none |
| Inline suppression (E-5) | `inline-suppression.sh` | not invoked | not invoked |

Root cause: `active.sh` was consumed only by `resolve.sh` + `measure-rubric.sh` (reporting);
`inline-suppression.sh` only by `migrate-rule.sh`. The config plane was a fully-built standalone
island. Crucially, the config-plane specs test their resolvers **in isolation** (E-3 → `active.sh`;
E-5 → `inline-suppression.sh`), so wiring the resolver *into* enforcement is purely additive.

## What this CL builds (E-1 severity/enable-disable + E-3 overrides → native enforcement)

Both `rubric/enforce-file.sh` (write-time, per-file) and `rubric/composite-audit.sh` (audit-time,
whole-tree) gain an **optional `--profile <profile.yaml>`**:

- **Absent → byte-identical** to before (every catalog rule active at native severity). The config
  plane is purely additive; no existing verdict can move. Verified: the no-`--profile` baseline
  stays red on a privileged pod, `rules_checked=25`, no `profile=` marker.
- **Present →** the effective per-file config is resolved through the existing §2.5 resolver
  (`profiles/active.sh --tree generated-code-quality-standards --emit-resolved --for-file <file>`),
  whose output is `{rules:{<id>:{severity}}}`. The enforcer then, for this file:
  - **disables** any rule resolved to `off`/`false`/`0` (skips its detector + excludes from grading),
  - **forces the grade** for `error` (block) / `warn` (advisory), and
  - leaves rules the profile does not mention at their native grade.
- The summary surfaces `profile=<name> disabled=<n>` when a profile is active.
- **Robust by construction:** a missing or malformed profile falls back to native enforcement (the
  resolver call is guarded; unparseable output → empty effective map → no-op), never crashing the
  gate.

The ordering the operator described is now explicit per file: resolve effective §16 config → for
each *enabled* rule, route by its stored 4-axis label / detector (§28) → enforce → grade by the
resolved severity. The 4-axis labeling stays upstream (assigned at ingest, stored on the rule);
config resolution is the per-file gate in front of enforcement.

## Why no regression (and the gate that enforces it)

- Existing config-plane specs drive the resolvers in isolation → untouched.
- Existing enforcement specs assert the all-active baseline; default-preserving threading keeps
  those verdicts byte-identical.
- New disable/override/fallback behavior is covered only by the 10 new `cl505-*` specs.
- The full eval suite is the hard commit gate.

## Scope boundary (what is deliberately NOT in this CL)

- **E-5 inline suppression** threading into enforcement is the next gated CL.
- The **external-tool path** (`composite-dispatch` → eslint/checkov/...) is not given per-rule CTP
  severity, by design: those tools enforce their *own* rule sets and carry their own config; CTP's
  per-rule config plane governs CTP's 118 rules, which run on the native detector path. This is a
  boundary, not a gap.
- No ESLint-engine reimplementation (cascading-config resolver / JS RuleTester) — that invented
  surface remains out of scope per §16's lightweight-parity choice.

## Spec sketch (`cl505-*`, 10 specs, tool-independent)

no-profile native baseline unchanged · `off` disables a rule · E-3 override glob match disables ·
profile omitting the rule keeps it enforced · E-3 override glob non-match keeps it enforced ·
summary surfaces profile+disabled · whole-tree audit honors the profile · whole-tree audit baseline
unchanged · unreadable profile safe-fallback · malformed profile safe-fallback.
