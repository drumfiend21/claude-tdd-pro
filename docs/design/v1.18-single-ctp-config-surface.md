# The single CTP rule-configuration surface (design)

> **Status: design / proposal.** Specifies ONE user-facing, ESLint-like configuration surface that
> makes every scraped CTP rule configurable in one consistent pattern, in an easy-to-find location,
> and projects that one file onto all 80 enforcement tools. **Anti-drift note:** the *schema* and
> *rule-addressing* are NOT invented here — they are the existing §2.5 profile system + G-3 rule IDs.
> What this devises is (a) the discoverable single entry point, (b) the scaffolding that makes every
> scraped rule findable, and (c) the universal projection onto every tool. New surface (the root
> file name, the `init` scaffolder, the projection) is proposed for approval + an append-only
> amendment before building. Builds on `v1.18-eslint-config-to-tool-config-map.md`.

## 1. What already exists (reuse, do not reinvent)

- **§2.5 profile system** is already the singular ESLint-pattern config:
  ```yaml
  extends: [ rubric:recommended, google:tsguide, <profile> ]   # presets / per-source / compose
  rules:                                                        # ESLint severity + options, by rule ID
    g-ts-001: error
    g-ts-002: ["error", { allow_with_comment: false }]
    g-react-008: ["warn", { budget_kb: 250 }]
  overrides:                                                    # ESLint glob overrides
    - files: ["**/*.test.ts"]
      rules: { g-ts-001: off }
  exclude_sources: [ google:* ]
  ```
- **G-3 rule IDs** address every scraped rule with a stable, namespaced key: `g-*` (plugin-shipped),
  `g-universal-*` (cross-cutting), `<operator-id>-*` (operator), `<plugin-id>/*` (community).
- **`userConfig.profile`** selects the active profile; **`_meta/INDEX.md`** + `rule-id-namespacing.md`
  are the browsable rule catalog; **E-1** `/doctor --explain <rule-id>` shows resolution order.
- **§28.44** already threads this config into the *native* enforcement path (resolve→enforce→grade).

So the "consistent, singular, CTP-specific, ESLint-like pattern" **is `§2.5`**. The three real gaps
are: it isn't surfaced as one easy-to-find file; every scraped rule isn't trivially *findable* to
toggle; and it doesn't yet project onto the *routed-tool* path.

## 2. The single, easy-to-find entry point

**Proposed canonical file: `ctp.config.yaml` at the repository root** — discoverable exactly like
`eslint.config.js` / `.eslintrc`. It **is a §2.5 profile document** (identical schema; zero new
vocabulary). Resolution precedence (most-specific wins, all via the existing resolver):

```
ctp.config.yaml (repo root)              ← the file a user edits; THE config
   └ extends → userConfig.profile (named profiles/<name>.yaml)
        └ extends → profiles/standard.yaml (ships; extends: [rubric:recommended])
             └ rubric:recommended (built-in default set)
```

If `ctp.config.yaml` is absent, behaviour is exactly today's (`userConfig.profile` → default) — so
the entry point is additive and default-preserving. One file, one place, one pattern.

## 3. Making every scraped rule findable & configurable

A config is only "easy" if the user can discover the ~hundreds of rule IDs to toggle. Three
affordances, all generated from the live corpus (no hand-maintenance):

1. **`ctp config init`** scaffolds `ctp.config.yaml` containing **every active rule at its default
   severity, commented** — each line annotated with the rule's source, its enforcing tool, and its
   docs URL. The ESLint-equivalent of a fully-expanded generated config:
   ```yaml
   rules:
     # g-k8s-no-privileged-container  [src: cncf/k8s-hardening]  [tool: checkov,kubescape]  (default: error)
     g-k8s-no-privileged-container: error
     # g-ts-no-explicit-any           [src: google/tsguide]      [tool: eslint,semgrep]      (default: warn)
     g-ts-no-explicit-any: warn
     # ... every scraped rule, grouped by source namespace ...
   ```
2. **`ctp config print [--for-file <path>]`** — the effective resolved config (extends + overrides
   applied), the `--print-config` analogue. Reuses `profiles/resolve.sh --for-file`.
3. **`_meta/INDEX.md`** (G-1) stays the browsable catalog; `/doctor --explain <id>` (E-1) shows why
   a rule resolved to its severity/source.

Every rule scraped from any source (§28 pipeline) lands in `generated-code-quality-standards/` with
a G-3 ID, so it is **automatically** addressable in `ctp.config.yaml` the moment it is ingested — no
per-rule config wiring needed.

## 4. The one pattern, projected onto all 80 tools

The single config drives every tool through the **two-layer translator** (from the config-to-tool
map). The user edits only `ctp.config.yaml`; CTP owns all per-tool config.

- **Layer 1 — universal post-hoc SARIF (correctness, all 80 tools).** CTP resolves the effective
  config per file, runs the routed tools, maps each SARIF finding back to its originating CTP rule
  (rule-id correlation table per tool), then **drops disabled/suppressed findings and regrades by the
  effective severity**. Works for every paradigm — including tools with no native severity/disable
  (vuln scanners, validators) — because grading happens on CTP's SARIF bus, not in the tool.
- **Layer 2 — native-config emitters (optimization, per paradigm).** From the same single config CTP
  *generates* each tool's native config (`ruff` `per-file-ignores`/`select`, `rubocop` `Enabled`,
  `golangci` `exclude-rules`, `checkov` `--skip-check`, `eslint` overrides, `semgrep` severities,
  inline directives) so disabled checks don't even run and the tool's own report stays clean. These
  generated files are **CTP-owned artifacts, never hand-edited** — preserving "consistent & singular."
- **Carve-outs (unchanged from the map):** per-rule **options** payloads live per-binding in
  `enforced_by[].config` (no cross-tool transform — inherent); **engine-local** settings
  (`env`/`globals`/`parser`/`settings`) are scoped to ESLint-family bindings only and are absent from
  the cross-tool surface.

```
ctp.config.yaml ──resolve(§2.5)──▶ effective per-file config ──▶ for each routed tool:
        (one file)                                              ├─ Layer 2: emit native config (skip disabled)
                                                                └─ run ──▶ SARIF ──▶ Layer 1: back-map to rule,
                                                                                      drop disabled/suppressed,
                                                                                      regrade by severity ──▶ verdict
```

## 5. Why this satisfies every requirement

| Requirement | How |
|---|---|
| All scraped rules configurable | every rule gets a G-3 ID at ingest → addressable in `rules:` automatically |
| Easy-to-find location | `ctp.config.yaml` at repo root, like `.eslintrc`; `ctp config init` scaffolds it |
| Consistent & singular CTP pattern | one schema (§2.5): `extends` / `rules` / `overrides`; user edits ONE file |
| Works with ALL enforcement tooling | two-layer translator: universal post-hoc SARIF (all 80) + generated native configs |
| User never juggles per-tool configs | per-tool configs are CTP-generated artifacts, never hand-edited |

## 6. Feasibility & scope

**Feasible**, building almost entirely on existing surface. New surface to approve + amend (append-
only) before building: (1) root `ctp.config.yaml` as the canonical §2.5 entry + resolution
precedence; (2) `ctp config init` / `ctp config print` scaffolder/printer; (3) the two-layer
projection onto the routed-tool path (Layer 1 is the §28.44 routed-path gap). The §2.5 schema, G-3
addressing, `userConfig.profile`, and native-path threading (§28.44) already exist and are unchanged.

**Proposed build sequence (each additive, default-preserving, full-suite-gated):**
1. `profiles/standard.yaml` (ships; `extends: [rubric:recommended]`) + root `ctp.config.yaml`
   resolution precedence (currently `standard.yaml` is referenced by §21 but absent).
2. `ctp config init` scaffolder (every rule + source + tool, generated) and `ctp config print`.
3. Layer 1 post-hoc SARIF back-mapping + severity/disable/suppression on `composite-dispatch`
   (closes the §28.44 routed-path boundary).
4. Layer 2 native-config emitters, per paradigm, highest-traffic tools first.
5. Per-binding `enforced_by[].config` options (C3) surfaced in the scaffolder.

All carve-outs and per-paradigm transforms are inherited verbatim from
`v1.18-eslint-config-to-tool-config-map.md`.
