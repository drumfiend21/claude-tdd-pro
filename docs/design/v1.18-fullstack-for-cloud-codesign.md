# v1.18 — Full-stack-for-cloud co-design: the two build flows inform each other (2026-06-30)

Operator directive: *"Build for cloud. The two development flows should inform each other."* — the
application-code build flow (Node backend / React frontend / database / services) and the IaC/cloud
build flow must be coupled, each informing the other, not run as disconnected legs.

## The gap

CTP has two build flows: **application code** (O-12 scaffolds + the W-7/W-8 TDD loop) and **IaC/cloud**
(S-29 `cloud-build`). They both trace to the same S-50 decision, but nothing LINKED them: the app's
components did not declare the cloud infra they need, and the infra did not declare the app it serves.

## The coupling (composition, no new feature ID)

`commands/codesign-build.sh` composes S-50 (decision) + S-52/S-53 (full-stack design pillars) + S-29
(IaC build units) + O-12 (app scaffolds). From the S-33 technical-requirements + a target cloud platform
it derives, for one decision, BOTH:

- **application build units** — `backend-api` (api/integration pillars), `frontend` (frontend/edge),
  `database` (data/storage), `realtime-service` (realtime), `auth-service` (identity) — each declaring
  the cloud infrastructure it **requires**;
- **infrastructure build units** — platform-native (`aws`: ecs-fargate / rds / cloudfront /
  api-gateway-websocket / secrets-manager; `gcp`: cloud-run / cloud-sql / cloud-cdn / …; `azure`:
  container-apps / azure-sql / azure-cdn / …) — each declaring the application component it **serves**;

and **reconciles** them: `reconciled=true` iff every app unit's required infra is present and every infra
unit serves an app unit. This is the bidirectional "inform each other" link — the application is built
FOR the cloud, and the cloud is provisioned FOR the application, from one design. The plan traces to the
S-50 `decision_id`, so both legs (app scaffold + IaC build unit) build from the same decision.
