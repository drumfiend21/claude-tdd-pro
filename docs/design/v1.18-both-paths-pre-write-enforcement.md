# v1.18 — Both-paths pre-write enforcement: IaC + full-stack rule sets govern in memory (2026-06-30)

Operator directive (the standing thread): both development paths' rule sets must govern everything CTP
generates, at every phase — including **in memory, before the write** — not just config/IaC. Uses the
§28.63 development-path tags as the partition.

## The gap

§28.60 governs proposed content before write, but the native `enforce-file` (and thus the pre-write
governor) scoped to **config/markup/IaC/prose** kinds and delegated application code (`.ts/.js/.py/…`)
to ESLint via `lint-on-save` (post-write). So the **full-stack rule set did not govern app-code
generation in memory** — only the IaC-path set did.

## The change

- **`enforce-file.sh` gains `--include-app-code`** (opt-in; default byte-identical). When set, a
  full-stack rule with no manifest glob derives its glob from its `linguist_aliases` (any language:
  typescript→`*.ts`, python→`*.py`, go→`*.go`, …), so enforce-file natively enforces the g-ts/g-react/
  g-node/g-python/… rule set. The native detectors (`no-any.sh`, `naked-throw.sh`, …) are deterministic
  and tool-independent (no ESLint required).
- **`enforce-file.sh` gains `--single-file-gate`**: detectors needing the whole tree (sibling test
  files etc. — `type-test-coverage.sh`) are not decidable on one proposed file in an isolated scratch,
  so they are skipped at the per-file write/pre-write gate (audit-time keeps them — the
  write-time-pragmatic vs audit-time-strict distinction).
- **`enforce-standards-pre-write.sh` (§28.60) extended** to the app-code kinds: it reconstructs the
  proposed content and runs `enforce-file --single-file-gate [--include-app-code]` in memory. So now
  **both rule sets govern before write**: IaC/config/prose via the IaC+cross-cutting set, app code via
  the full-stack+cross-cutting set. A P0/P1 violation on either path denies the write.

## Result (verified)

- A proposed `.ts` with `: any` → **denied** in memory by `g-ts-001/002` (full-stack), before save.
- A proposed `.tf` with `0.0.0.0/0` → **denied** by `g-aws-no-unrestricted-ingress` (IaC).
- Clean app code (incl. exported symbols) → allowed (tree-context coverage rule skipped pre-write).
- Both deterministic + tool-independent (native detectors); the post-write hook + audit-time behavior
  is unchanged (the new flags are opt-in).

Together with the design-phase enforcement (the architect already runs enforce-file on every ADR,
cl487) and audit-time (`composite-audit`), both development-path rule sets now govern everything CTP
generates at design, in-memory-before-write, write, and audit phases.
