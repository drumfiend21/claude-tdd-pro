# v1.29 — Repo-wide `audit` command + total rule enforcement (DESIGN)

**Status: DESIGN — operator-directed, backlogged for build.** Architecture pointer: §35.
Builds on the whole-repo enforcement assessment (this session) and the CL-564.i scraping
fixes. Records the operator's requirements verbatim so they drive the build.

## 1. The invariant (goal)

**Every file in the repo has every rule its four-axis tags mark applicable actually
enforced — native detector OR 3rd-party tool — at all three moments: write-to-file time,
generation time, and audit time. Zero silent skips, zero vacuous greens.** This is the
architecture's own §28.57 (universal native enforcer) + §29.6 (one byte-identical
enforcement path), made real and total.

## 2. What checks rules today, and the gap (answer to operator Q "#3 — for tagging?")

Today's standing gates check **tagging / metadata completeness**, NOT per-file enforcement
coverage:

| Gate | Checks |
|---|---|
| `classify-path.sh --audit` | every rule tagged `iac`/`fullstack`/`both` (`unpathed=0`) |
| `config-sync --check` | every rule has projectable tool options |
| `source-citations` | every rule cites a source URL |
| `applicable-rules` / `rules-verified` | a handoff *carries* rule lists |

**None verifies that every file actually had its applicable rules enforced.** The operator's
intuition is correct: "always checked" today means tagging + handoff-metadata, not
enforcement coverage. This design adds the missing enforcement-coverage guarantee.

## 3. The `audit` command (centerpiece)

One command runs the checking of code against the rules. Runnable on the **entire repo**
(default) or on **any set of folders / filenames / glob patterns**, ESLint-style.

### 3.1 Wildcard file matching (globbing) — operator spec, verbatim

Core purpose: flexible file/directory targeting using standard path-matching wildcards
(`node-glob` syntax).

Supported syntax:
- `*` — matches zero or more characters in a single path segment.
- `?` — matches exactly one character in a single path segment.
- `**` — matches zero or more directories recursively.
- `!` — negates the pattern (excludes previously matched files).

Execution flow:
1. **Input:** accept an array of include patterns and an optional array of exclude patterns.
2. **Expansion:** resolve all patterns against the local project filesystem.
3. **Filtering:** apply inclusion patterns first, then subtract any files matching
   exclusion/negation patterns.
4. **Output:** a flat list of absolute file paths for processing.

Default with no patterns = the entire repository.

### 3.2 What audit does per matched file

For each file in the resolved flat list: compute its four-axis applicability, enforce
**every applicable rule** (native detector and/or routed 3rd-party tool, with the native
fallback when a tool is absent — §28.56/§28.57), fold verdicts. **Fail (non-zero exit) on
`red > 0` OR `not_enforced > 0`** over the matched set — no silent skip counts as a pass.

### 3.3 Integration-test-first (operator-directed)

When auditing the **entire repo**, audit **looks for integration tests first**. Then:
- If integration tests exist and pass → proceed with rule enforcement.
- If they are **absent or failing** → audit must **establish them**: create the repo's
  integration-test scaffold, and **leverage the tests to drive generation of excellent code**
  (test-first), then **pin them** so they lock established functionality against regressions
  and newly-introduced bugs. (This composes with the existing TDD-Guard / regression-pin
  machinery; it does not replace it.)

## 4. Generation-time total enforcement (operator-directed)

When the plugin **generates content**, all applicable rules must be enforced so **all
generated content conforms** to the rules relevant to it per four-axis tagging. This extends
§28.60 (govern-before-write: enforce proposed content in memory before it is saved) to the
**total** applicable set — the same selection + enforcement the `audit` command uses, applied
to generated content before it lands.

**Explicit scope (operator-directed): this covers every generation surface, not just
application code.** Generated content for **cloud architecture** (design docs / ADRs / IaC),
**development** (app + service code), **deployment**, and **DevOps** (CI/CD pipelines,
provider manifests, container/orchestration config, IaC modules) must each be enforced
against the rules its four-axis tags mark applicable — IaC-dialect rules on `.tf`/CloudFormation/
K8s manifests, pipeline rules on CI configs, prose/architectural rules on design ADRs, and
language rules on code — before that content is written. No generation surface is exempt: the
IaC-native-glob fix (§5) is a hard prerequisite so cloud/DevOps rules are natively enforceable
when their 3rd-party tool is absent, closing the biggest generation-time hole (today cloud/IaC
content relies entirely on 3rd-party tools with no native fallback).

## 5. Enforcement-correctness fixes (from the assessment — prerequisites for §§3–4)

The `audit` and generation guarantees are only true if native selection is total. The
assessment found these leaks; they are fixed first:

- **App-code native enforcement** only fires with `--include-app-code`; passed only by
  `enforce-write-time.sh`. **Add it** at the three leaking native paths:
  `enforce-standards-on-save.sh`, `composite-audit.sh`, and `composite-dispatch.sh`
  `native_fallback()` — routed through the §29.6 shared primitive where the file is a single
  proposed file, direct-flag where the walk is whole-tree.
- **IaC-tagged rules have no native glob path** in `enforce-file.sh` (language-only map).
  **Add an `iac_aliases`→glob mapping** (terraform→`*.tf`, cloudformation→`*.yaml/*.json`
  templates, k8s→manifests, …) so IaC rules are natively selectable when the IaC tool is
  absent.
- **Write-time allowlist + `not_enforced`:** widen the pre-write extension allowlist to the
  audit file-universe, and make `not_enforced` (exit 3) a surfaced/blocking condition rather
  than a silent pass.

## 6. Shell scripts (research recorded; build DEFERRED per operator)

`.sh` is the repo's largest kind (445 files) with **no applicable rule and no detector**
today. Tooling research: **ShellCheck** (de-facto standard; catches quoting, unset vars, the
bash-3.2 portability traps this repo already tracks) + **shfmt** (formatting).
npm-accessible: `shellcheck` (npm binary wrapper), `mvdan-sh`/`sh-syntax` (pure-JS parsers);
non-npm standard: `bashate` (pip). Build path when scheduled: route ShellCheck as the 3rd-party
tool + native fallback; source shell rules from ShellCheck wiki / Google Shell Style Guide
URLs (dogfoods the scrape→tag→enforce chain on the plugin's own code). **Deferred — operator
acknowledged "later work on the shell scripts."**
