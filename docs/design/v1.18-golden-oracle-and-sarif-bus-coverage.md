# v1.18 — Golden-master Katas oracle + SARIF-bus tool-soup coverage (2026-06-30)

Closes the two integration-coverage gaps identified during the §28.54 post-build coverage audit
(inverted dependency-closure analysis: 460/480 substrate files exercised; the committed golden masters
were *committed but never used as test oracles*, and no single test asserted architect/scaffold output
flowing through the routed tool-soup to the SARIF bus). Conformance/integration hardening only — **no new
feature ID, no new §2.X contract.** Extends §27.27 (cloud-architect output-conformance, `cl465-*`),
§28.14 (generative-function E2E, `cl473-*`), and §28.28/§28.29 (the SARIF 2.1.0 bus).

## Gap 1 — Golden-master Katas oracle (`cl522-golden-01..10`)

`cl465-conformance-*` already oracled the live `business-translate.sh` output against the committed
`fullstack-international-aws-requirements.json`. It left the two *other* committed Katas scenarios as
dead reference data. This family wires them in as oracles:

- **Healthcare-booking-HIPAA scenario** (a 2nd distinct vision): live `business-translate.sh
  --profile healthcare-booking-hipaa-profile.json` meets-or-exceeds the committed
  `healthcare-booking-hipaa-requirements.json` (no-regression); the committed golden is itself fully
  cited (`needs_grounding=[]`, every concern `grounding=grounded`); every live decision is cited; and
  the regulated-data inputs tailor the design with `audit_logging` + `mfa` + `strong_consistency`.
- **Multicloud boundary responses**: live `aws-boundary.sh` on the committed `multicloud/aws-handoff.json`
  is `validated` and its `build_units` match the committed `multicloud/aws-boundary-response.json`
  oracle; `gcp-boundary.sh` / `azure-boundary.sh` validate their committed handoffs with the correct
  `platform`; all three handoff goldens are committed + platform-tagged; the committed AWS
  boundary-response is validated with every normalized recommendation grounded.
- **Tailoring (not a template)**: the HIPAA and full-stack-AWS visions yield *distinct* concern sets
  (HIPAA-only: `audit_logging`/`mfa`/`strong_consistency`; AWS-only: `cdn`/`multi_region`/
  `eventual_consistency`), proving the oracle covers ≥2 genuinely different scenarios (§28.14).

## Gap 2 — Tool-soup SARIF-bus over architect/scaffold output (`cl523-sarif-01..10`)

Proves the composite engine routes architect/scaffold-generated content through whatever routed
3rd-party tools are present **and** the CTP-native detectors, normalizing every verdict to the
§28.28/§28.29 SARIF 2.1.0 bus — at write-time and audit-time. **Tool-independent** (green on a toolless
container): the routed path is exercised with `RUN_TOOL_FORCE_ABSENT=1` so the engine behaves
identically everywhere; with-findings normalization is proven with crafted SARIF docs (the bus is the
unit under test):

- scaffolded IaC routes through `composite-dispatch.sh` to one normalized per-file verdict;
- the bus merges a routed tool's output into a single SARIF 2.1.0 log; an error-level finding turns it
  red; a no-blocking stream stays green; an identical finding reported by two docs is deduplicated;
  a `--strict` (audit-time) bus counts a warning as blocking; the per-tool result counts are reported;
- a routed tool emits valid SARIF 2.1.0 even when its binary is absent (tool-independent stream);
- an architect-generated ADR tree audits through `composite-audit.sh` to one normalized tree verdict;
  and an architect ADR flows `run-tool.sh` → bus → normalized verdict.
