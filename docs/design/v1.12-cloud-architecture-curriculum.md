## 1. Goal (English-first, per PRIMARY RULESET §1)

Give the plugin a **standing watch** on the world's authoritative cloud-architecture guidance so that
an operator practising at DARPA/DoD and xAI/SpaceX tiers always designs against the *current* state of
the art — not a frozen snapshot. Concretely:

1. A curated, operator-editable catalog of tier-1 cloud-architecture sources (AWS / Azure / GCP
   Well-Architected, cloud design patterns, IaC, security/resilience/scale, DoD cloud guidance).
2. A poller that re-fetches each source on **any frequency the operator specifies, expressed in
   milliseconds** (`<N>ms`/`<N>s`/`<N>m`/`<N>h`), **defaulting to `daily`**, cheaply (conditional GETs),
   and **only while a Claude Code session is active** (in-use semantics).
3. A **continuous-education digest** that turns the raw "what changed upstream" stream into an
   operator-readable brief organised by the six Well-Architected pillars and the six curriculum phases —
   so "best practices and new technologies" surface as a readable delta, not a diff dump.

Everything ships **wide-and-deep TDD-first** (specs in `evals/pending/` before any substrate), then
development to satisfy the specs, per the per-CL workflow loop in `CLAUDE.md`.

### §27.1 Why these gaps are architecture work

(a) **Configurable-frequency / in-use polling (S-20)** is architectural because the cadence grammar
(`daily` vs `<N>ms`) and the "in-use" semantics (a fetch fires on a wall-clock interval **only while a
session is active**, detected via the §2.13 active-flow stack) are invariants other features depend on:
§2.8 provenance `freshness_at_generation`, S-16 freshness gate, H-1/H-12 cost telemetry all read the
resolved cadence. Distinct from S-13/S-17 (which are *calendar* first-use-of-day refresh); S-20 is the
sub-day, session-scoped poller.

(b) **Conditional-GET layer (S-21)** is the cost-defensibility prerequisite that makes (a) safe: without
RFC 7232 `304 Not Modified` handling, millisecond polling would re-download and re-diff stable
whitepapers on every tick. The persistence shape (`etag`/`last_modified` alongside `content_hash`) and
the 304-skips-pipeline semantics are contract-level (§2.29), consumed by H-12 cost rollup (304:200 ratio).

(c) **`FETCH-FREQUENCIES.yaml` (S-22)** is the operator-facing resolution surface for `any-frequency` and
per-registry defaults across all three registries (STANDARDS-URLS, PR-SOURCES, COMPLIANCE-URLS). Resolution
order (per-source override > registry default > global default `daily`) is an invariant, hence architectural.

(d) **Cloud-architecture domain seed (S-23)** is architecturally a *data + catalog* extension of S-1, but
it introduces four source namespaces (`aws`, `azure`, `gcp`, `hashicorp`) that the G-9 auto-scaffold and
the §2.21 source-folder contract must accept; the namespace set is part of the architecture surface
(cf. G-1's enumerated 14 namespaces).

(e) **Continuous-education digest (S-24)** is genuinely new consumer surface, distinct from S-5
`/standards-diff` (which is a per-source Adopt/Defer/Reject decision log) and S-9
`STANDARDS-CONFORMANCE.md` (which is rule-conformance). S-24 is *pedagogy-shaped*: it answers "what
should I, the architect, go learn this week?" organised by pillar and curriculum phase — the education
framing the request asks for, not a conformance report.

### §27.2 Boundary discipline preserved

- **S-20 honors §2.17 live-freshness and §2.13 active-flow.** In-use polling never runs without an active
  session on the active-flow stack; offline → falls back to `offline-cached` per §2.8, never blocks.
- **S-21 conditional GET is a fetcher optimisation, not a freshness bypass.** A `304` updates the
  `standards-last-fetch/<id>.timestamp` (freshness proof) but does not alter `content_hash` or
  re-trigger S-5 diff. Paywalled HEAD-only sources keep current behavior (conditional GET moot).
- **S-22 does not fork the cadence state machine.** `FETCH-FREQUENCIES.yaml` resolves a value that feeds
  the **same** `fetch_frequency` field in §2.6/§2.12/§2.19; it is operator config, not a new contract field.
- **S-23 uses G-9 auto-scaffold and §2.21 source-folder contract unchanged.** Each new source maps to one
  registry entry, one `<namespace>/<id>.yaml` with a populated `source:` header; no bespoke directory.
- **S-24 reads existing telemetry; writes a derived artifact.** No new data collected vs S-5/S-10; the
  digest is a serialization/rollup over the existing diff stream (cf. §2.24 audit-pack reasoning).

### §27.3 Cross-cutting contracts introduced

**§2.28 Configurable-frequency in-use polling contract.** A source's `fetch_frequency` (in §2.6 standards,
§2.12 PR-corpus, §2.19 compliance) accepts EITHER a calendar token
`daily | weekly | monthly | quarterly | on-demand` (existing, unchanged) OR a sub-day interval matching
`^[0-9]+(ms|s|m|h)$` (e.g. `300000ms`, `30s`, `5m`, `2h`) OR the shorthand `any-frequency`. Default when
unset is `daily`. Sub-day intervals fire **only while a Claude Code session is active**, detected by a
non-empty §2.13 active-flow stack; with no active session, the source degrades to its calendar default and
`freshness_at_generation` reports `offline-cached`. `any-frequency` resolves via §2.22 (S-22)
`FETCH-FREQUENCIES.yaml` in resolution order: per-source override → per-registry default → global default
`daily`. `/doctor` displays `next-fetch-eta` per source; H-1/H-12 record per-source fetch cost. The
resolved cadence is recorded in the §2.8 provenance manifest `standards_state.<source>`.

**§2.29 Conditional-GET freshness-economy contract.** Each non-paywalled fetcher (S-2 `html-anchor`,
`markdown-headers`, `rfc-style`; L-2 GitHub endpoints; the compliance fetcher) persists the upstream
`etag` and `last_modified` response headers alongside `content_hash` in
`.claude-tdd-pro/standards-last-fetch/<id>.json`. Subsequent fetches send `If-None-Match: <etag>` and/or
`If-Modified-Since: <last_modified>`. On `304 Not Modified`: update the freshness timestamp, increment the
304 counter, and skip the downstream parse/diff/hash-compare pipeline. On `200`: run the full S-2 pipeline
and refresh stored headers. H-12 cost rollup records the per-source `304:200` ratio so the operator can
see how cheap the daily-or-finer posture actually is. Paywalled/HEAD-only sources are exempt (HEAD already
cheap). Conditional GET is a fetch optimisation only — it never advances `content_hash` or suppresses an
S-5 diff when content actually changed.

### §27.5 Out of scope for §27

- **A hosted/managed curriculum service.** S-24 ships the local digest; no remote dashboard.
- **Auto-adopting upstream changes into rules.** S-24 surfaces deltas for human Adopt/Defer/Reject via the
  existing S-5 decision log; it never silently rewrites rules.
- **Non-cloud curriculum domains.** The seed is cloud-architecture; other domains are future S-domain seeds.
- **Vendor SDK / API-rate-limit negotiation.** Conditional GET (§2.29) is best-effort RFC 7232; sources that
  do not honor `ETag`/`Last-Modified` simply fall back to full fetch (no special-casing per vendor).
- **Sub-millisecond / hard-real-time cadence.** Grammar floor is `1ms`; the scheduler is best-effort
  wall-clock, not a real-time timer.

## 6. TDD shape — what "wide and deep" means per feature

Per `CLAUDE.md` Step 1/Step 2: **10 specs per feature** (active + pending combined), each asserting a
*distinct behavior slice* (no pattern-cloning, drift mechanism #5), hermetic (runner tmpdir + setup
arrays), state-asserting (explicit `expect.exit_code`, meaningful `stderr_contains`), behavior-named
(≥20 chars, no opaque IDs). Specs are authored into `evals/pending/...` first; substrate is built to
satisfy them; then `probe-feature`/`promote-pending` (with `cl<N>-` prefix) moves them to `evals/specs/`.

### 10-behavior spec sketches (the TDD that comes *first*)

**S-20 configurable-frequency polling scheduler** — `evals/pending/s/s-20-configurable-frequency-polling-scheduler/`
1. scheduler parses `300000ms` cadence and reports next-fetch-eta as 300000ms from last fetch
2. scheduler parses `30s` / `5m` / `2h` interval grammar and normalises to milliseconds
3. unset `fetch_frequency` resolves to the default `daily` cadence
4. sub-day cadence fires only when the active-flow stack is non-empty (session active)
5. with an empty active-flow stack, source degrades to its calendar default (no fire)
6. invalid cadence token (`5x`, `-1ms`, `0`) is rejected with a non-zero exit and message
7. `any-frequency` defers resolution to FETCH-FREQUENCIES.yaml (S-22 handoff)
8. `/doctor` surface lists `next-fetch-eta` per source
9. resolved cadence is written into the provenance `standards_state.<source>` block
10. scheduler is idempotent — re-run before the interval elapses performs no fetch

**S-21 conditional-GET fetcher layer** — `evals/pending/s/s-21-conditional-get-fetcher-layer/`
1. successful 200 persists `etag` and `last_modified` alongside `content_hash`
2. subsequent fetch sends `If-None-Match` with the stored etag
3. subsequent fetch sends `If-Modified-Since` with the stored last_modified
4. a 304 response updates the freshness timestamp without changing `content_hash`
5. a 304 response skips the parse/diff pipeline (no new S-5 decision emitted)
6. a 200 with changed body advances `content_hash` and triggers the S-5 diff
7. paywalled/HEAD-only source is exempt — no conditional headers sent
8. missing/blank upstream etag falls back to full fetch (no crash)
9. H-12 cost telemetry records the per-source 304:200 ratio
10. malformed stored header file is tolerated (treated as cold fetch)

**S-22 FETCH-FREQUENCIES registry** — `evals/pending/s/s-22-fetch-frequencies-registry/`
1. absent file → global default `daily` for every source
2. per-registry default key overrides global for that registry's sources
3. per-source override key wins over per-registry default
4. resolution order is exactly override > registry-default > global-default
5. `any-frequency` in a source entry triggers FETCH-FREQUENCIES lookup
6. a `<N>ms` value in the file validates against the §2.28 grammar
7. malformed file value is rejected with a clear message (no silent daily fallback masking error)
8. file applies symmetrically to STANDARDS / PR-SOURCES / COMPLIANCE registries
9. `/doctor` reflects the resolved per-source frequency from the file
10. editing the file is picked up on next resolution without restart

**S-23 cloud-architecture standards domain seed** — `evals/pending/s/s-23-cloud-architecture-standards-domain-seed/`
1. all 12 seed sources present in `standards/sources.yaml` with required §2.6 fields
2. each seed source is `tier: 1`, `class: published-standard`, `applies_to: [cloud-architecture]`
3. namespaces `aws`, `azure`, `gcp`, `hashicorp` source-folders exist (G-9 scaffold)
4. DoD/NIST sources map to existing `us-government` namespace (no new namespace)
5. CNCF source maps to existing `linux-foundation` namespace
6. each seed source has a populated `source:` header per §2.21 in its scaffolded file
7. each seed source declares a valid `fetch_frequency` (default `daily`)
8. each seed source carries a `curriculum_phase` 1–6 tag for S-24 grouping
9. `/standards-add` can add a tier-2 community cloud source without disturbing seeds
10. removing a seed via `/standards-remove` archives its folder file to `_archived/`

**S-24 continuous education digest** — `evals/pending/s/s-24-continuous-cloud-architecture-education-digest/`
1. digest groups deltas by the six Well-Architected pillars
2. digest groups deltas by the six curriculum phases
3. a source whose `content_hash` changed appears in the digest's "best practices updated" section
4. a delta tagged `new_technology` appears in the "new technologies" section
5. no-change run produces an empty-but-valid digest (header + "no changes since <ts>")
6. digest writes both `<utc>.md` and `<utc>.json` to `standards/curriculum-digest/`
7. JSON digest validates against its schema (pillars[], phases[], new_technology[])
8. `--dry-run` previews the digest without writing files (§2.14)
9. digest cites source ids + fetched_at for every delta (traceability)
10. digest is reproducible — identical inputs produce identical content (sorted keys)

**§2.28 contract** — `evals/pending/cross-cutting/2-28-configurable-frequency-polling-contract/` (10 specs
asserting grammar acceptance/rejection, default-daily, in-use gating, any-frequency handoff, provenance record).

**§2.29 contract** — `evals/pending/cross-cutting/2-29-conditional-get-freshness-economy-contract/` (10 specs
asserting header persistence, 304 semantics, pipeline-skip, paywalled exemption, cost-ratio record).

Total new specs: **7 features × 10 = 70 pending specs.** Active regression suite (58) stays untouched
