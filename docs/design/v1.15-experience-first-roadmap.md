# v1.15 — Experience-First Build Roadmap: "Vision in, optimized implementation out"

**Status:** planned (2026-06-08). Registers feature S-46 (§27.17) and an experience-first delivery ordering over the already-registered S-26..S-45.

## North-star user journey (the thing every milestone serves)

> A business-minded or creative founder with no technical expertise states their app **vision** in plain language. The plugin guides them through **every** cloud-architecture decision in business terms, then **implements** it across the available platform APIs — optimized to be **cost-effective**, **performance-optimized**, **customer-centric**, and **shareholder-centric** (good for the software company building with the plugin).

Two consequences shape the build order:
1. **Experience-first (walking skeleton).** Build the thinnest *end-to-end* path (vision → guided decision → option → ADR → build → enforce) before deepening any one stage. The whole journey must run early, then get richer — not one perfect stage at a time.
2. **Objectives are first-class.** Cost / performance / customer / shareholder value are an explicit, grounded, scored dimension (S-46), woven from the skeleton onward — not a late add-on.

## The four optimization objectives → grounding (all sources already secured)

| Objective | Means | Grounded in |
|---|---|---|
| Cost-effective | right-sizing, managed services, spend visibility | `finops-framework`, AWS WAF Cost pillar |
| Performance-optimized | latency, throughput, scaling | AWS WAF Performance Efficiency pillar |
| Customer-centric | availability/RTO-RPO, latency, reliability | `aws-reliability-pillar`, `aws-rpo-rto-targets` |
| Shareholder-centric (company value) | cost, time-to-market, maintainability, low lock-in | `finops-framework`, `google-eng-practices`, operational-excellence |

## New capability required by the journey

### S-46 — Objective-weighted optimization & option scoring
`commands/optimize-options.sh`. Reads the S-34 options + the business-profile and scores/ranks each option against the four grounded objectives, producing a ranked `option-scoring.json` with per-objective scores + grounded rationale. Weights derive from the profile (e.g., `budget_posture=cost-first` lifts the cost weight; `criticality=mission-critical` lifts customer/availability) and are overridable. cite-or-decline: an objective with no backing source is `needs_grounding`. This is what turns "here are 3 options" into "here is the option that best serves your cost + customer + shareholder priorities, and why."

## Delivery milestones (each ships as TDD CLs; ordering respects dependencies)

**M0 — Walking skeleton: vision → enforced artifact (end-to-end, thin).**
Build **S-34** (recommendation composer; start single-best, structured for multi) and the **S-36** orchestrator entry that accepts a free-text *vision* and drives the chain through already-built S-32/S-33 → S-26 review → S-28 ADR → S-29 build → S-30 enforce. *Outcome:* a founder's vision produces one grounded, enforced build unit. The journey runs.

**M1 — Guided architect (decisions in business language).**
**S-35** plain-language explainer + full **S-36** orchestrator (vision→structured intake bootstrap, `next_question`, plain-language summary) + **S-34** emits 2–4 options with trade-offs. *Outcome:* a beginner is genuinely guided through all decisions, in business terms, with options.

**M2 — Optimization objectives first-class.**
**S-46** objective-weighted scoring. *Outcome:* options are ranked by cost / performance / customer / shareholder value, each with grounded rationale — the user's optimization mandate, delivered.

**M3 — Domain depth: data, queues, distributed systems.**
**S-37** sources → **S-38** data-aware intake → **S-39** data/distributed translation. *Outcome:* guidance now covers databases, queues/messaging, and distributed-systems patterns, not just compute.

**M4 — Implement across all platform APIs.**
**S-41** boundary contract → **S-42/S-43/S-44** AWS/Azure/GCP boundaries (wrap WA Tool / Advisor / Recommender APIs, injected/normalized). *Outcome:* "implement with all available APIs"; cross-cloud options become real.

**M5 — Full implementation toolchain.**
**S-45** toolchain advisor (containers/GitOps/CI-CD/migrations/messaging/observability/policy-as-code/testing/FinOps). *Outcome:* the rest of what's needed to actually run in production is recommended + scaffolded.

**M6 — Closed loop & hardening.**
Wire S-30 enforcement into `/doctor` + CI; the Learn loop (S-24/S-25) keeps guidance current; FinOps cost feedback + customer/shareholder KPIs surfaced back into S-46. *Outcome:* the system stays optimal over time.

## Dependency notes
- S-34 is the keystone (M0); S-46 (M2) and the boundaries (M4) all enrich S-34's options.
- S-37 precedes S-38/S-39 (sources before the mappings that cite them).
- S-41 precedes S-42/S-43/S-44 (contract before the boundaries that implement it).
- Every milestone keeps the suite green and the constitution additive.

## Non-goals
- No live cloud credentials in the hermetic suite (native APIs injected as fixtures).
- No reproduction of book sources; ground in free primary catalogs.
- The agent conducts the conversation; the scripts supply the grounded, deterministic artifacts.
