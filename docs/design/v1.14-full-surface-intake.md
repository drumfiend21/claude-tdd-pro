# v1.14 — Full-surface requirements intake (S-57 / §2.35 / §30)

**Inbound:** GCTP proposal **P-12** (relayed 2026-07-04). GCTP filed it as "§27.16 Full-Surface Intake" —
but **§27.16 already exists** in CTP's architecture ("Layered multi-cloud advisor"), so that label is a
collision. CTP owns its own decomposition (drift mechanisms #1/#2): P-12 is a *requirements input*, and
this amendment assigns it CTP-correct coordinates — **feature S-57, contract §2.35, section §30**.

## The gap (input-side mirror of P-11 / §29)

§29 (P-11) made CTP's **output** complete against the full 42-namespace / 118-rule surface. But the
**intake** that feeds the chain — S-32 `business-intake.sh` — asks a **fixed universal-9 questionnaire**
regardless of workload. So most of the 42 namespaces are grounded from *defaults / inference*, never from
a fact the founder actually stated. Full-surface *output* still rests on a *narrow input*.

## The fix — S-57 full-surface intake (a new command, composing S-32)

Following the P-11 precedent (S-56 was a *new* `full-surface-consult.sh`, not a chain rewrite), S-57 is a
new command `commands/full-surface-intake.sh` that **composes** S-32:

1. **Workload classifier** (`standards/business-intake-workload-classifier.yaml`) — classifies the founder's
   plain-language vision into `workload_types`, each putting a set of the literal aggregator **namespaces**
   in scope. Signal-substring match; a `baseline-quality` type is `always: true`.
2. **Per-namespace probe groups** (`standards/business-intake-question-bank.yaml`) — for each in-scope
   namespace with a probe group, targeted business-language follow-ups, each citing an **existing** catalog
   `source_id` (grounding is real; ~most sources already exist per GCTP's manifest correction).
3. **v1.1 `business-profile.json`** — a **strict additive superset** of the v1.0 profile:
   - `answers` — the universal 9, **mirrored unchanged** (universal-stays-universal);
   - `workload_classification` = `{ workload_types, namespaces, activated_probe_namespaces }`;
   - `probes` = `{ <namespace>: { <probe_key>: <answer> } }`;
   - `grounded_in` — a **strict superset** of the v1.0 value (universal sources ∪ answered-probe sources);
   - `grounded_in_namespaces` — namespaces grounded from a *gathered* probe answer.
4. **Schema** `schemas/business-profile.schema.json` — `oneOf` on `schema_version`: v1.0 and v1.1 both
   valid; v1.1 additionally requires the three new blocks. Validated by CTP's own
   `rubric/detectors/lib/validate-json-schema.js` (draft-2020-12 subset).

S-57 shells `business-intake.sh` for the universal layer + its validation — so v1.0 behavior is untouched
and **back-compat is guaranteed by construction**. An invalid universal answer still fails loud (exit 2,
delegated); an out-of-enum probe answer fails loud (exit 2); intake stays incomplete (exit 1) until every
activated probe is answered.

## §2.35 — the intake boundary contract (3 surfaces)

Per GCTP's boundary description, and **authoritative from CTP** (GCTP reconciles its PENDING validator to
this shape):

1. **`business-profile.json` schema** — v1.0 ∪ v1.1 as above; v1.1 is a strict additive superset.
2. **`--list-questions` JSON** — `{ universal_source, probe_groups: { <ns>: [ …questions… ] } }` for the
   activated namespaces; `--classify` emits `{ workload_classification }`.
3. **`source_id ↔ namespace` traceability** — every probe cites an existing catalog `source_id`; every
   `grounded_in_namespaces` entry is an `activated_probe_namespaces` entry backed by ≥1 answered probe.

**Additivity invariants (all tested):** universal 9 stay universal · v1.0 profiles still validate ·
`grounded_in` is a strict superset · no rule relaxed.

## §30.1 — design-engine consumption (the input→design close-out, 2026-07-04)

Originally (CL-546) the consumers were left untouched — the v1.1 profile carried `probes` but nothing
consumed them, so committed postures did not yet steer option choice. GCTP's KATA audit flagged this as the
open half. CL-547 closes it (extending S-33/S-34; no new feature ID):

- **`business-translate.sh`** reads `probes.<namespace>` and adds a GROUNDED concern per committed posture
  (cited by the probe `source_id`): `owasp_threat_posture` → threat_modeling + penetration_testing;
  `slsa_build_level` → provenance_attestation; `sbom_generation` → sbom_generation; `react_accessibility_target`
  → accessibility_conformance; `aws_region_strategy=multi-region` → multi_region; `aws_cost_guardrails` →
  cost_guardrails; `k8s_multitenancy=multi-tenant` → namespace_isolation; `hashicorp_state_backend` →
  remote_state_locking; `jwt_*` → short_lived_tokens; `observability_signal_depth=full-tracing` →
  distributed_tracing. Emits `probes_consumed=<n>`. The grounding-verification catalog additionally loads
  `eo-security-sources.yaml` + `sources.yaml` so `slsa-framework` / `wcag-2-2` resolve as grounded (additive).
- **`architect-recommend.sh`** lets a decisive commitment move the pick: multi-region upgrades a balanced
  default to the most-resilient option; a hard cost cap pulls to cost-optimized. Emits `probes_consumed=<n>`.
- **Back-compat by construction:** both paths gate on `probes` presence; a v1.0 profile is unchanged
  (`probes_consumed=0`, same concerns, same pick).

`architect-session.sh` still needs no change — it composes translate + recommend, which now consume probes.

## §30.2 — precise cloud classification + IaC probe coverage + transparency (2026-07-04)

GCTP's KATA readiness audit surfaced a coverage gap: the classifier put 8 IaC namespaces in scope on a
generic `terraform` signal (`hashicorp, aws, azure, gcp, cfn, ansible, iam, security-governance`) but the
question bank probed only 4 — `azure/gcp/cfn/ansible` were **silently** in-scope-but-unprobed. Two problems:
(1) over-scoping (a bare Terraform vision pulled every cloud into scope), and (2) silent gaps.

CL-548 fixes both (refines the S-57 corpora; no new feature ID):

- **Precise cloud classification.** `iac-cloud` now scopes only provider-agnostic namespaces
  (`hashicorp, iam, security-governance`). Dedicated types fire on cloud-specific signals:
  `aws-platform`→`aws`, `azure-platform`→`azure`, `gcp-platform`→`gcp`, `cloudformation`→`cfn`,
  `config-management`→`ansible`. An AWS-only kata is probed for `aws`+`cfn`, not Azure/GCP.
- **IaC probe coverage.** Added grounded probe groups for `azure`, `gcp`, `cfn` (region strategy, landing
  zone / project structure, stack policy / drift detection). `business-translate` consumes them
  (`azure/gcp_region_strategy=multi-region`→multi_region; `cfn_stack_policy=protected`→stack_protection).
  `ansible` has no grounded founder-decision source in the catalogs, so it is left unprobed and REPORTED
  (not silently dropped).
- **Coverage transparency.** `full-surface-intake` computes `unprobed_in_scope` and reports it on
  `--classify`, the run marker, and the persisted `workload_classification` block — the intake mirror of
  "no rule silently unenforced". Remaining unprobed namespaces (CI-platform alternatives `gha/circleci/
  jenkins/glci/azdo`, `compose/mesh/md/web-vitals`) carry no distinct founder commitment and are grounded
  at output time by §29.

8 specs (`cl548-cover-01..08`). Suite 4931→4939.

## §30.3 — word-boundary signal matching (classifier precision fix, 2026-07-05)

GCTP's kata pre-flight (running `--classify` on the real Certifiable, Inc. vision prose before a live
`/consult`) caught a real bug: `full-surface-intake` matched classifier signals by **substring**, so short
signals collided with ordinary English:

- `aks` (Azure Kubernetes Service) matched inside "le**aks**" / "content leaks" → spurious `azure-platform`
  + `container-orchestration` (and their probes: an AWS-only kata would have been asked Azure region-strategy
  questions — the exact hole §30.2 set out to close, reopened by the matcher);
- `ci` matched inside "**ci**rtification" / "certifi**c**ation" / "a**cc**reditation" → spurious `ci-cd`;
- `spa` matched inside "**spa**ce".

§30.2 fixed the *dispatch* (right types on right signals) but not the *matcher*. CL-549 fixes the matcher
(single-line change; no new feature ID):

```ruby
# before: hay.include?(sig)                                   # substring — collides
# after:  hay.match?(/(?<![a-z0-9])#{Regexp.escape(sig)}s?(?![a-z0-9])/)   # whole token (+ optional plural)
```

Alphanumeric boundaries so multi-word phrases (`amazon web services`) and internal punctuation (`ci/cd`)
still match; optional trailing `s` so plurals (`microservices`, `queues`, `containers`) still match. The
change can only *tighten* the classifier (fewer false-positive types), never loosen it. Verified: the real
AI-credentialing prose now classifies to `ai-governed` + `baseline-quality` only, while real `AKS` / `CI/CD`
tokens still fire.

8 specs (`cl549-precision-01..08`). Suite 4939→4947.
