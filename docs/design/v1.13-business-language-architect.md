# v1.13 — Business-Language Architect Advisory Layer

**Status:** designed (2026-06-08). Registered additively as architecture §27.15 (features S-32..S-36).
**Goal:** let an engineer who knows little about cloud act *through* the plugin as a competent architect, making decisions in **business language** rather than technical language — closing the gaps the current S-26..S-31 stack leaves open.

## 1. Problem statement (the gap)

The S-26..S-31 cloud-architect stack produces excellent *technical* artifacts (pillar reviews, ADRs, test-first IaC, grounded convention enforcement), but it assumes the user can already:

- translate a business need ("a HIPAA web app for 10k users that can't be down for more than an hour") into the technical concerns the review/build stages consume;
- answer the architect-grade questions nobody asked them (criticality, RTO/RPO tolerance, data sensitivity, compliance regime, budget-vs-uptime posture, scale);
- read a scaffold of `findings`/`trade_offs`/`risk_tier` and know what "least privilege" or "multi-AZ" *means for their situation*;
- decide, not just receive options.

This layer supplies the missing front-end: **elicit business inputs → translate to technical concerns → recommend grounded decisions → explain in plain language → orchestrate the whole flow** — while the existing S-26..S-31 substrate does the technical work and Claude (the agent) conducts the actual conversation using these structured, grounded artifacts.

## 2. Design principles (inherited, unchanged)

- **Grounded / cite-or-decline.** Every translation, recommendation, and glossary entry cites a catalog source (S-23/S-30/S-31) or is declined.
- **Deterministic substrate + agent front-end.** The scripts produce structured, reproducible JSON/MD (the questions to ask, the mappings, the recommendations, the plain-language explanations). The *conversation* is Claude reading and acting on those artifacts. We do not pretend the shell is conversational; we give the agent a grounded script to follow.
- **Composes, does not fork.** The orchestrator chains the existing S-26 (review), S-28 (ADR), S-29 (build), S-30 (enforce) — no reimplementation.
- **Test-first.** Each feature ships as one CL with 10 hermetic specs.

## 3. Feature decomposition (S-32..S-36, Phase S)

### S-32 — Business-language requirements intake
`commands/business-intake.sh`. A structured questionnaire that captures business inputs in plain terms and emits a `business-profile.json`. The question set IS the senior-architect follow-up list:
- `workload` (free text), `criticality` (mission-critical|important|experimental), `availability_tolerance` (none|minutes|hours|days → drives RTO/RPO), `data_sensitivity` (public|internal|confidential|regulated), `compliance_regime` (none|hipaa|pci|fedramp|il4|il5|gdpr), `scale` (small|medium|large|hyperscale), `budget_posture` (cost-first|balanced|uptime-first), `region_constraints` (free text).
*How:* fixed question schema with allowed-answer enums; validates each answer against its enum; reports `unanswered=<keys>` and `invalid=<key>` so the agent knows exactly what follow-ups to ask; `--answer key=value` (repeatable) or `--answers <json>`; writes `business-profile.json` only when complete (or `--partial`). Exit 0 complete / 1 incomplete / 2 invalid-answer.

### S-33 — Business-to-technical translation
`commands/business-translate.sh`. Maps a `business-profile.json` to technical/pillar concerns, each grounded. *How:* a deterministic mapping table (business answer → pillar concerns + grounding source). E.g. `criticality=mission-critical` + `availability_tolerance=minutes` → reliability: `multi_az`, `automated_failover`, `health_check`, `backup` (grounded in aws-well-architected / google-sre-book); `data_sensitivity=regulated` + `compliance_regime=hipaa` → security: `encryption_at_rest`, `encryption_in_transit`, `audit_logging`, `access_control` (grounded in nist-800-53 / aws-prescriptive-security). Output `technical-requirements.json` keyed by pillar, each concern carrying `{concern, driver (the business answer), source_id}`. This is the bridge into S-26/S-29.

### S-34 — Architect recommendation engine
`commands/architect-recommend.sh`. Turns the profile + technical requirements into **recommended decisions** (opinionated, not just concerns). *How:* a rule-based recommender keyed on the business profile; each recommendation is `{decision, pillar, driver, rationale, source_id, default}` — e.g. "Adopt multi-AZ deployment — because criticality=mission-critical and you can tolerate only minutes of downtime (AWS Well-Architected REL)." Recommendations are emitted in a shape directly consumable by S-28 (`--emit-adr-args`) and S-29 (`--emit-requirements`). Exit 0; `needs_grounding` when no source backs a concern.

### S-35 — Plain-language explainer
`commands/explain.sh`. A grounded glossary that translates technical terms and review/ADR findings back into business language. *How:* a term→plain-language table, each entry `{term, plain, why_it_matters, source_id}`; `--term <t>` explains one term; `--review <review.json>` annotates every pillar finding with a plain-language gloss; `--profile` tailors the wording to the captured business context. Unknown term → `unknown_term` (declines rather than inventing). This is what lets a beginner *understand* the output.

### S-36 — Guided architect session orchestrator
`commands/architect-session.sh`. The single entry point a beginner runs; chains the flow end-to-end. *How:* runs S-32 intake (and surfaces `unanswered` so the agent asks follow-ups), then on a complete profile runs S-33 translate → S-34 recommend → S-26 review → (optionally) S-28 ADRs + S-29 build units, and produces a `session.json` + a plain-language `session.md` summary via S-35. Emits `next_question=<key>` while incomplete (drives the conversation) and `session_complete=true` with the artifact paths when done. Composes existing commands; adds only the flow + the gap-driven question surface.

## 4. How each identified gap is closed

| Gap (from the analysis) | Closed by |
|---|---|
| "Won't ask smart follow-up questions" | S-32 question schema + S-36 `next_question`/`unanswered` surface |
| "You describe in business terms but it doesn't translate" | S-33 business→technical mapping (grounded) |
| "It scaffolds but doesn't decide" | S-34 recommendation engine (opinionated, grounded) |
| "You must interpret 'least privilege', 'multi-AZ'…" | S-35 plain-language explainer/glossary |
| "Doesn't tie it together conversationally" | S-36 orchestrator + agent front-end |

The remaining honest residue — deep judgment, politics, novel systems — stays with a human reviewer; the layer makes the 80% disciplined, grounded, and legible.

## 5. Ticket plan (one CL per feature, TDD, §20 weeks 43–47)

- **CL-A (S-32):** `business-intake.sh` + 10 specs (enum validation, unanswered/invalid surfacing, partial vs complete, profile emission, dry-run).
- **CL-B (S-33):** `business-translate.sh` + 10 specs (criticality→reliability concerns, regulated→security concerns, each concern grounded, compliance-regime mapping, profile-driven, cite-or-decline).
- **CL-C (S-34):** `architect-recommend.sh` + 10 specs (recommendation carries driver+rationale+source, mission-critical→multi-AZ, regulated→encryption, emits ADR args, emits build requirements, needs_grounding path).
- **CL-D (S-35):** `explain.sh` + 10 specs (term gloss, why-it-matters, grounded, annotate-review, unknown-term declines, profile-tailored).
- **CL-E (S-36):** `architect-session.sh` + 10 specs (drives next_question while incomplete, runs full chain on complete profile, composes S-26/S-28/S-29, plain-language summary via S-35, session artifact, dry-run).

Each CL follows the standard loop: Step 0 arch quote → Step 1 specs in `evals/pending/s/<id>-<label>/` → Step 0.5 fidelity gate → probe with `cl<N>-` prefix → full-suite verify → commit+push.

## 6. Non-goals
- No live LLM calls inside the shell (the agent is the conversational layer).
- No new ADR/IaC/convention schema (reuse S-28/S-29/S-30).
- No replacement of human review on high-stakes decisions.
